# 交易主链修复设计 Brief

## 用户与运行场景

- 用户为具备投资基础的个人用户或专业交易者，使用 1024 px 以上桌面浏览器。
- 当前账户、订单、成交均为仿真数据；行情与 K 线来自 PandaData。
- 用户已完成登录，进入交易工作区后需要在同一身份下创建或读取仿真账户并完成一次可追溯的仿真订单。

## 页面任务与主要对象

- T00 `/settings`：初始化和查看当前仿真账户。
- T02 `/markets`：选择 PandaData 支持的当前标的，并把选择传递给交易台。
- T03 `/desk`：围绕当前标的创建订单草稿，展示风险复核结果并提交。
- T07 `/orders`：查看提交后的订单状态、错误和可执行操作。
- T08 `/portfolio`：只展示后端账户、成交与持仓事实；不使用成本价伪装缺失行情。
- 主要对象依次为 `SimulationAccount`、`Instrument`、`OrderDraft`、`Order`、`Fill`。

## 信息层级与工作区区域

- 顶部：产品导航、仿真模式、PandaData 状态、上游时点与刷新状态。
- 左栏：当前标的或订单列表。
- 主栏：订单输入、草稿摘要、风险复核和唯一主操作。
- 右栏：当前账户、风险与证据；沿用工作区区域，不增加浮动聊天机器人。
- 底栏：行情数据状态或订单时间线；失败与过期状态必须可见。

## 数据源、频率、新鲜度与失败状态

- 行情与 K 线：后端 PandaData 适配器；前端轮询最高每秒一次，显示供应商时间、实际频率和新鲜度。
- 账户、草稿、订单和成交：FastAPI 仿真 API 与 PostgreSQL；明确标记“仿真”。
- 登录身份：JWT `sub` 是账户、工作区与订单的唯一 owner。
- 行情失败显示错误或 stale，不用成本价、默认报价或演示数据替代。
- 账户不存在显示初始化操作；订单请求失败显示失败，不显示为“暂无订单”。

## 主操作与启用条件

- 主操作为“提交仿真订单”。
- 仅在账户可用、标的受支持、输入有效、草稿已复核、风险无硬阻断且用户已确认摘要时启用。
- 重复提交使用幂等键；状态未知时禁止创建同一草稿的第二笔订单。

## 视觉与交互命题

- 视觉命题：浅色、紧凑、以主表和订单复核为中心的专业交易工作区，蓝色只用于选择与主操作。
- 内容计划：账户状态 → 标的与行情 → 订单输入 → 风险复核 → 提交结果与订单状态。
- 交互命题：栏目切换只替换相关区域；草稿阶段变化提供短暂状态反馈；遵守 `prefers-reduced-motion`。

## 影响范围

- 路由：`/settings`、`/markets`、`/desk`、`/orders`、`/portfolio`。
- 目标宽度：1440 px；最低支持 1024 px；低于 1024 px 显示明确桌面宽度提示。
- 本轮不实现实盘账户、券商授权、移动布局、自动交易或新增营销内容。

## 可验证成功标准

1. 两个登录用户不能读取或修改彼此的仿真资源。
2. 新用户可创建仿真账户，并完成 draft → review → confirm → submit。
3. 写入跨 HTTP 请求持久存在，失败事务回滚且会话关闭。
4. 行情或订单 API 失败时显示明确错误，不伪装为空列表或有效盈亏。
5. `/api/ready` 能识别页面实际依赖的行情能力，生产部署不会在核心依赖不可用时报告成功。
