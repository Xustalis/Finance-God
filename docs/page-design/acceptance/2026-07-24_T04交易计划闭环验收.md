# T04 交易计划闭环验收

> 验收对象：候选/组合偏离 → 版本化交易计划 → 审阅确认 → 精确版本绑定的仿真订单草稿。

## 1. 任务与信息架构

| 验收项 | 结果 | 证据 |
| --- | --- | --- |
| 页面首屏能识别用户任务、当前对象和主要状态 | 通过 | `/trade-plans/:planId` 首屏显示计划编号、版本、状态、来源和仿真标识；见 `output/playwright/t04-trade-plan-1440.png`。 |
| 每个区域只有一个明确职责 | 通过 | 7 列动作区只编辑动作，5 列 Inspector 只显示影响、警告和版本历史，底部操作条只承载保存与确认。 |
| 主操作唯一且结果可预测 | 通过 | 唯一主操作为“确认计划并生成草稿”；后端只追加确认版本并生成引用该精确版本的 `planned` 草稿。 |
| 页面与上一步、下一步对象保持连续 | 通过 | T02、T08 创建计划后进入详情；确认成功后展示可进入 T05 的草稿链接。 |
| 没有无功能价值的区域或卡片 | 通过 | 页面由工具栏、动作表、影响 Inspector、版本历史和共享 AI 侧栏组成，无装饰卡片。 |

## 3. 真实数据

| 验收项 | 结果 | 证据 |
| --- | --- | --- |
| 市场数据来自 PandaData，不含硬编码行情 | 通过 | `TradePlanService` 通过后端行情应用读取报价并写入版本快照；缺行情直接失败，不生成演示值。 |
| 显示来源、供应商时间、实际频率和新鲜度 | 通过 | 页面工具栏与数据状态显示 provider、provider time、frequency、freshness。 |
| 页面刷新间隔与数据频率分别表达 | 不适用 | T04 不轮询；行情只在创建和保存版本时取快照，页面明确展示该快照的实际频率。 |
| 请求失败不会回退到模拟行情 | 通过 | 行情配置或请求失败时后端返回显式错误，`TradePlanService` 不含模拟报价分支。 |
| 真实行情与仿真账户/订单明确分离 | 通过 | PandaData 标为真实行情；账户、持仓、计划和草稿均显示“仿真”。 |
| 浏览器不可访问 PandaData 凭据 | 通过 | 浏览器只调用同源 `/api/trade-plans`；行情凭据只由后端适配器读取。 |

## 10. 最终结论

- 验收结论：有条件通过。T04 产品闭环、版本不变量、幂等草稿生成和桌面页面均已验证；全仓前端检查和真实 PandaData 冒烟仍有外部阻断。
- 未通过项目：当前全仓前端静态检查；真实 PandaData 请求冒烟测试。
- 已知限制：必须在具备 PandaData 凭据/连通性的部署环境复跑真实行情冒烟；并发 T10 `evidence.spec.ts` 现有 4 个类型错误及 3 个 evidence drawer 运行失败，与 T04 目标测试无关。
- 复现命令或测试入口：
  - `cd backend && .venv/bin/python -m pytest -q`
  - `cd frontend && npm run type-check && npm run lint && npm run build`
  - `cd frontend && npm test -- --run src/tests/trade-plan.spec.ts src/tests/watchlist-candidates.spec.ts src/tests/app-ai-shell.spec.ts`
  - `curl -i 'http://127.0.0.1:8010/market/quotes?symbols=000001.SZ'`
