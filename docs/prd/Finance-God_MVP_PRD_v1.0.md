## 1. 文档目的与产品决策

### 1.1 文档目的

本 PRD 将 Finance-God 定义为一个从理解用户、形成量化画像、构造投资授权，到由 Multi-Agent Workflow 完成投研、因子、策略、验证、报告、组合构造与复盘的完整产品流程。它为产品、设计、前后端、量化、数据、AI、安全与合规团队定义 MVP 的交付边界、规则、接口、验收标准与待决策项。

本文件描述产品和技术需求，不构成投资、证券、税务或法律建议。任何实盘、市场接入、资产范围和用户地区的上线许可，均须在对应范围内单独完成合规、数据授权与安全评审。

### 1.1.1 当前实现与 MVP 目标态

截至本版基线日期，仓库中的可运行实现集中在 Python Multi-Agent 执行层：已接入固定版本的统一运行时，具备统一请求与结果契约、能力路由、资源和执行权限门禁、并发编排以及离线场景实验。用户认知引擎、投资授权书、组合构造、仿真交易、风控控制台、完整 Web/移动端产品和经纪商接入尚未形成可交付系统。

因此，本文使用以下状态口径：

| 状态 | 定义 |
|---|---|
| 当前已实现 | 能在当前仓库代码中定位，并由测试、可复现实验或运行说明验证的能力 |
| MVP 目标态 | 本 PRD 要求在 P0 交付并通过验收的能力 |
| 后续候选 | P1 或完成专项安全、数据、合规和账户评审后才可进入交付范围的能力 |

除非条目明确标注为“当前已实现”，本文中的“必须”“支持”“系统可”等需求用语均描述 MVP 目标态，不应据此对外宣称能力已经上线。实现进度必须以代码、测试结果和独立交付状态记录为准。

### 1.2 一句话定位

**Finance-God 是普通人的首个 AI 理财顾问：先理解其完整财务状况、人生阶段、风险偏好与当前心智状态，再用可审计的多智能体协作系统构造、验证、执行与持续维护组合。**

产品的唯一闭环为：理解用户的财务全景与人生阶段 → 量化画像与心智状态 → 投资授权书 → Planner 任务规划 → Data、Research、Factor 协作 → Strategy 生成 → Backtest 验证 → Report 解释 → 组合构造 → 仿真或授权执行 → 风险监控与复盘 → 更新画像和授权书。

产品不以传统问卷加人工解释作为核心体验。它应将用户资料、行为和授权转化为可校正的理解，再以通俗语言说明“发生了什么、为什么重要、可做什么和不能做什么”，并允许用户展开查看数据、规则与证据。

### 1.3 双引擎产品架构

| 层级 | 负责的问题 | 核心输出 | 不负责 |
|---|---|---|---|
| 用户认知引擎 | 用户是谁，能承受什么，想达成什么，约束是什么。 | 量化画像、陈述与行为揭示的风险、心智状态快照、风险预算、约束集、目标优先级、投资授权书。 | 选择某一只资产或绕过用户约束。 |
| Multi-Agent Workflow 多智能体协作系统 | 如何将复杂投研问题拆解、协作、验证、解释和持续优化。 | 任务计划、数据包、研究备忘录、因子研究、策略候选、回测报告、用户报告与可验证的下游输入。 | 由任何单一 Agent 决定交易、修改用户风险边界或突破硬风控。 |
| 产品组合层 | 如何把两类能力呈现为用户可理解、可控制的长期产品。 | 当前计划、目标组合、交易状态、解释、复盘和控制面板。 | 将复杂 Agent 过程暴露为不可理解的黑箱。 |

### 1.3.1 动态信号的使用边界

| 信号 | 可以影响 | 不能影响 |
|---|---|---|
| 用户心智状态与行为偏差 | 解释方式、提醒优先级、冷静期、附加确认、复盘任务和是否暂停新的订单意图。 | 自动提高风险、修改长期目标、放宽授权、改变硬约束或在未确认情况下交易。 |
| 市场情绪与事件环境 | Data、Research、Factor 与 Strategy 的任务优先级、风险提示、数据可用状态和是否暂停执行。 | 绕过授权、在数据不足时强制生成组合或将短期情绪当作收益保证。 |

### 1.4 MVP 必须达成的结果

1. 用户可在约 12 分钟内完成建档，形成覆盖财务全景、人生阶段、目标、约束与风险的量化用户画像。
2. 用户可确认或修正陈述风险、行为揭示风险和当前心智状态之间的差异；低置信度状态不会被静默当作交易依据。
3. 用户可保存一份投资授权书，明确目标、时间、风险预算、流动性、资产范围、集中度上限和 Agent 自主级别。
4. 系统可为已授权用户生成可解释的目标组合和调仓计划，所有建议均可追溯到画像、授权、策略、市场环境与数据版本。
5. Agent 可在模拟账户中完成研究、组合构造、下单、成交回报、风控拦截与复盘的完整交易生命周期。
6. 任何交易意图在触发硬约束、数据异常、授权失效、策略失效、心智冷静期或风险熔断时必须被阻断，并向用户和运营方显式说明原因。
7. 实盘能力只在完成账户授权、经纪商接入、可用范围配置和安全评审后启用；MVP 必须保留完整的实盘接入契约，但不将其作为默认可用能力。

### 1.5 MVP 关键产品决策

| 决策 | MVP 定义 | 原因 |
|---|---|---|
| 顾问体验 | 面向非专业用户，默认用通俗语言说明财务现状、建议、限制与下一步；用户可展开查看数据、规则和证据。 | 让 AI 的理解和决策可以被普通人理解、校正和控制。 |
| 用户画像 | 使用结构化问答、行为数据和持续复盘生成版本化的量化画像；每一维都有来源与置信度。 | 避免把一次问卷或模型猜测当作永久事实。 |
| 心智状态 | 将陈述风险、行为揭示风险与当前心智状态分开记录；心智信号仅能触发解释、冷静期、确认或复盘。 | 避免将短期焦虑、贪婪或模型误判直接变成资产配置指令。 |
| 市场环境 | 市场情绪、事件和数据质量由独立 Agent 输出，必须带来源、时点、适用范围与置信度。 | 防止短期噪声或陈旧数据驱动策略。 |
| 风险边界 | 风险预算、资金用途、流动性、资产范围和集中度限制由用户授权书与规则引擎共同定义。 | 用户约束是交易 Agent 不可逾越的边界。 |
| 多智能体协作 | Planner、Data、Research、Factor、Strategy、Backtest 与 Report 通过版本化产物通信；组合、风控和执行在工作流外独立校验。 | 让复杂问题的规划、证据、策略、验证和解释可分工、反馈与追溯。 |
| 组合构造 | 目标组合由策略和优化器生成，必须满足用户授权书和风险规则；AI 可解释但不能跳过校验。 | 将智能能力置于可验证的约束系统内。 |
| MVP 交易 | P0 支持真实市场数据上的仿真交易与完整订单生命周期；实盘仅保留集成和授权设计。 | 先验证决策与执行链路，再扩大资金与合规风险。 |
| 实盘自主级别 | 实盘必须基于用户明确的授权策略，可配置逐笔确认、调仓批次确认和预授权自动执行。 | 让用户控制自动化程度，而不是二元地开启或关闭 Agent。 |
| 资产范围 | P0 仅支持经过主数据确认的 ETF 与公募基金；个股、衍生品、杠杆和加密资产不在公开 MVP。 | 降低建模、数据、流动性和合规复杂度。 |
| 表达边界 | 向用户展示策略、风险、概率和不确定性，不承诺收益或隐藏亏损可能。 | 建立长期可信的交易产品体验。 |

## 2. 用户、场景与非目标

### 2.1 目标用户

| 用户 | 典型特征 | 核心任务 | MVP 价值 |
|---|---|---|---|
| 普通理财用户 | 有收入、储蓄或负债，但没有专业投顾服务，也难以把生活目标转成投资计划。 | 先看懂自己的财务状况和可承受范围。 | 获得一个可解释、可校正且不以专业术语为前提的理财入口。 |
| 长期投资新手 | 有稳定储蓄但缺少投资框架，对交易自动化既好奇又谨慎。 | 把生活目标转成清晰、可控的投资授权。 | 先理解用户，再以低自主级别体验仿真交易。 |
| 自主投资者 | 已持有基金或 ETF，手工调仓成本高且难以坚持纪律。 | 看懂现有组合并授权 Agent 执行规则化策略。 | 在自己定义的边界内获得组合构造和复盘能力。 |
| 策略验证者 | 愿意比较多种资产配置与再平衡方法。 | 观察策略依据、执行过程、风险和偏离。 | 获得可复放的策略和仿真执行记录。 |

### 2.2 核心用户故事

- 作为长期投资者，我希望系统理解我的现金需求、目标日期、损失承受能力和偏好，而不是只让我选一个风险标签。
- 作为普通理财用户，我希望先看懂收入、支出、资产、负债、保障和人生阶段如何共同影响我的选择，而不是被直接推向某个产品。
- 作为用户，我希望看到量化画像如何影响我的风险预算、资产范围和组合权重，并能修正错误的理解。
- 作为非专业用户，我希望先获得通俗解释，再按需要查看数据、规则和风险，而不是只能阅读术语和图表。
- 作为选择自动化的用户，我希望给 Agent 一份明确授权，决定它能否研究、提议调仓、模拟执行或代表我提交实盘订单。
- 作为已有持仓的用户，我希望 Agent 在提出或执行交易前先说明组合影响，而不是单独推荐某个标的。
- 作为审慎用户，我希望随时暂停 Agent、收紧授权或撤销未来订单，并完整查看历史决定与执行记录。

### 2.3 非目标与使用限制

MVP 不服务日内高频交易、保证收益、黑箱策略跟单、杠杆、期权、加密资产、私募、结构性产品、税务筹划或法律意见。

MVP 不允许 Agent 绕过用户授权书、风险规则、市场状态检查或范围控制。未经启用的地区、资产、账户和经纪商不提供实盘执行能力。产品不保存经纪商密码；所有交易接入必须采用受支持的授权机制。

## 3. 范围、版本与成功指标

### 3.1 当前仓库能力矩阵

下表描述当前代码和可复现实验能够证明的能力，不表示完整 MVP 已经交付：

| 能力域 | 当前事实 | 与 MVP 目标的差距 |
|---|---|---|
| Multi-Agent 运行时 | 已接入 `verifolio-research-runtime 0.2.0`；目录包含 43 个 Agent 定义，其中 TradingAgents 12 个、FinRobot 19 个、QuantSkills 12 个 | 现有 Agent 目录是通用研究与量化能力池；交易域目标态另增加 `financegod:planner`，合计 44 个 Agent |
| 运行契约与路由 | 已使用统一 `AgentRequest` / `AgentRun`，支持能力匹配、执行档位、资源门禁、逐次授权、显式点名与并发上限 | 尚未实现面向产品业务对象的 TaskPlan、版本化质量门和完整失败恢复策略 |
| 场景工作流 | 已实现公司研究、市场环境、组合压力、策略验证 4 类正常工作流，以及只读复核、数据质量诊断 2 类替代工作流；用户暂停和硬风控可直接产生阻断产物 | 当前选择条件是内存中的 `WorkflowContext` 字段，尚未连接真实用户、授权、组合、市场数据质量或风控服务 |
| 证据与产物 | 阶段结果可转换为 `EvidenceRecord` 传递给后续阶段，并输出 `WorkflowRun`、结构化 JSON 与人读 Markdown | 尚无持久化、版本仓库、审计查询 API 或面向用户的证据页面 |
| 模型与数据适配 | 已提供 OpenAI Responses-compatible 模型配置；PandaData 和 FinRobot Metrics 仅在显式开启并满足资源门禁时使用；离线实验使用确定性适配器 | 离线适配器不是生产市场数据，当前仓库没有持续数据同步、主数据、质量治理或供应商 SLA |
| 用户与投资产品层 | 未实现用户画像、心智状态、投资授权书、持仓诊断、目标组合、交易计划、仿真账户、订单、成交、复盘或经纪商连接服务 | 这些仍是 P0 目标态，需要按本 PRD 的对象、接口、规则和验收场景建设 |
| 应用与基础设施 | 当前仓库是 Python 包、脚本、测试和后端运行时基线；没有可运行的 FastAPI 产品 API、React 应用、数据库迁移、PostgreSQL/Redis 或容器化应用栈 | 需在后续阶段建立产品应用、持久化、身份安全、可观测性和部署基线 |

基线验证结果：`tests.test_workflows` 的 9 项测试与 `tests.test_multi_agent` 的 5 项测试全部通过，Python 编译检查通过。

### 3.2 P0 范围

| 能力域 | P0 交付 |
|---|---|
| 量化用户画像 | 收入、支出、资产、负债、流动性、保障信息、人生阶段、目标、期限、本位币、经验、风险、行为偏好和限制条件的结构化建档。 |
| 用户认知模型 | 画像维度、数据来源、置信度、完整度、版本、冲突提示和变更影响分析。 |
| 心智状态与干预 | 经同意采集的行为和对话信号、状态快照、认知偏差提示、冷静期、确认和复盘任务。 |
| 投资授权书 | 目标优先级、风险预算、现金边界、资产范围、集中度限制、再平衡频率、交易自主级别和有效期。 |
| 持仓诊断 | 手动输入与 CSV 导入；资产标准化、未解析项、暴露、集中度、重叠和目标偏离。 |
| Multi-Agent Workflow | Planner 任务规划、Data 数据包、Research 信息分析、Factor 因子研究、Strategy 策略候选、Backtest 验证和 Report 研究报告。 |
| 策略与组合构造 | 在授权约束下生成策略候选、目标组合、交易意图和可解释的调仓计划。 |
| 仿真交易 | 模拟账户、订单生命周期、成交模型、费用与滑点假设、持仓与现金更新。 |
| 风控与控制台 | 事前校验、订单拦截、熔断、Agent 暂停、授权撤销、事件告警和审计。 |
| 复盘 | 定期比较画像、授权、目标组合、实际持仓、执行质量和策略表现。 |

### 3.3 P0 明确不包含

- 公开可用的实盘下单、资金划转、保证金、融资融券或衍生品交易；
- 自动提升用户风险等级、自动扩大资产范围或静默修改风险限制；
- 未验证来源的市场数据、无法确认身份的资产或不完整持仓上的正向交易结论；
- 基于短期涨跌、社交情绪或单一 Agent 判断的强制交易；
- 未经单独同意的可穿戴、外部社交或设备数据采集；
- 将心智状态、浏览行为或对话情绪自动转换为风险提升、策略改写或交易执行；
- 将仿真表现表述为实际收益、承诺收益或未来业绩。

### 3.4 P1 候选范围

在完成市场、账户、数据、安全与合规评审后，可评估受支持经纪商的实盘接入、用户授权的批次调仓、资产组合扩展、多币种账户、税务敏感再平衡和持牌人工顾问协作。

### 3.5 成功指标

| 指标 | MVP 目标 | 定义 |
|---|---:|---|
| 画像完成率 | 不低于 60% | 开始建档至达到可生成授权书最低门槛的用户比例。 |
| 画像校正率 | 不低于 20% | 查看画像解释后主动确认或修正至少一个维度的用户比例。 |
| 心智状态确认率 | 不低于 50% | 收到状态或偏差提示后确认、修正或关闭该提示的用户比例。 |
| 授权书保存率 | 不低于 45% | 完成画像后保存首个有效授权书的比例。 |
| 组合解释查看率 | 不低于 60% | 生成目标组合后查看目标、约束和风险解释的比例。 |
| 仿真交易闭环率 | 不低于 50% | 启动策略后完成至少一轮生成、校验、下单、成交与复盘的比例。 |
| 风控阻断正确率 | 100% | 触发硬约束、数据异常或授权失效时，订单不得进入执行队列。 |
| 心智干预越权率 | 0% | 心智状态未被用来自动提升风险、放宽授权或提交订单。 |
| 可追溯覆盖率 | 100% | 目标组合、订单意图和成交结果均能关联到版本化输入与事件。 |
| 30 日复盘率 | 不低于 30% | 保存授权书并启动仿真策略的用户中，30 天内完成一次复盘的比例。 |

## 4. 产品体验与信息架构

### 4.1 主流程

1. 用户选择本位币、可用地区和产品范围。
2. 用户完成建档；系统先生成财务健康摘要，再生成可编辑的量化用户画像、完整度、置信度和可选的心智状态快照。
3. 用户确认或修正画像与心智状态，生成投资授权书并选择 Agent 自主级别。
4. 用户录入持仓或从空组合开始，查看当前组合与授权目标的差异。
5. Planner 将用户目标、授权和当前状态拆成可验收任务；Data、Research 与 Factor 产出可追溯的证据和研究结果。
6. Strategy 生成候选策略；Backtest 验证其历史表现、假设、局限与失效条件；Report 将通过验证的内容解释给用户。
7. 组合服务和风控服务校验每个目标权重和订单意图；用户查看组合理由、回测限制、风险与调仓计划。
8. 执行服务在模拟账户创建、路由和跟踪订单；实盘接入启用后按授权模式执行同一流程。
9. 产品持续监控风险、策略漂移、授权到期和画像变化，并在复盘中更新下一周期输入。

### 4.2 信息架构

| 一级模块 | 二级页面 | 用户目的 |
|---|---|---|
| 首页 | 心智与组合状态、市场环境、风险告警、待确认事项、复盘任务 | 知道 Agent 是否在安全边界内运行。 |
| 认识我 | 财务健康、人生阶段、建档、量化画像、心智状态、偏差提示、画像解释、变更历史 | 校正系统对自己的理解。 |
| 我的授权 | 投资授权书、风险预算、资产范围、自主级别、有效期 | 决定 Agent 可以做什么。 |
| 我的组合 | 当前持仓、目标组合、暴露、偏离、调仓计划 | 理解从现在到目标的路径。 |
| Agent 中心 | Planner、Data、Research、Factor、Strategy、Backtest、Report、组合、执行、风控的状态与产物 | 查看分工、通信、反馈、验证和阻断。 |
| 交易 | 模拟账户、订单、成交、费用、滑点、实盘连接状态 | 跟踪执行，不把执行当作黑箱。 |
| 复盘与信任 | 复盘、数据来源、审计日志、提醒、导出、暂停与删除 | 管理数据、授权和自动化。 |

|---|---|---|

## 5. 详细功能需求

### 5.1 量化用户画像

量化用户画像由目标、财务约束、风险、行为与偏好五类维度构成。风险必须分为陈述风险、行为揭示风险与当前心智状态；三者均记录来源、采集时间、置信度和版本。系统可使用规则与模型提出画像或偏差推断，但任何低置信度、冲突或过期项必须向用户显式展示，不得作为提高风险、放宽授权或开启更高自主级别的依据。

| 维度 | 最低字段 | 输出作用 |
|---|---|---|
| 目标 | 目标名称、金额或区间、目标日期、优先级。 | 决定投资期限和资金分桶。 |
| 财务全景 | 本位币、收入与支出区间、主要资产与负债、应急资金、近期现金需求、保障信息和人生阶段。 | 决定现金边界、目标优先级和风险承受能力。 |
| 风险 | 可承受损失、波动容忍、经验、偏好、风险冲突。 | 决定风险预算和资产类别上限。 |
| 行为 | 复盘频率、对回撤的反应、自主偏好、信息偏好。 | 决定解释方式、提醒强度和默认自主级别。 |
| 心智状态 | 经同意采集的交易和应用行为、对话反馈、用户主动日记、状态趋势与认知偏差提示。 | 决定是否提示、启动冷静期、增加确认或安排复盘；不直接修改长期授权。 |
| 限制 | 地区、产品禁忌、集中度、可持续偏好、合规限制。 | 限制资产池、策略和订单。 |

#### 5.1.1 数据输入源

高隐私数据源必须采用单独、可撤回的明确同意；所有来源均在用户侧显示启用状态、最后采集时间和用途。

| 数据源 | 采集方式 | 更新频率 | 隐私级别 |
|---|---|---|---|
| 初始画像 | 智能问卷（自适应 12–15 题） | 一次性 + 季度复评 | 中 |
| 交易行为 | 交易记录、持仓变动、止损设置 | 实时 | 高 |
| APP 行为 | 登录频率、停留时长、页面浏览模式 | 实时 | 中 |
| 对话文本 | 与 AI 对话的语气、用词、情绪表达 | 每次交互 | 中 |
| 情绪日记 | 用户主动标注心态（可选功能） | 用户触发 | 低 |
| 外部信号 | 可穿戴数据，例如心率、睡眠；仅限用户授权 | 可选 | 高 |

| ID | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| UP-01 | 系统要求本位币、至少一个目标、目标期限和近期现金需求后才生成个人化画像。 | P0 | 缺失关键项时仅展示通用示例和缺项说明。 |
| UP-02 | 系统分别计算风险承受能力与风险偏好。 | P0 | 两者冲突时显示原因，并采用更审慎的结果作为硬边界。 |
| UP-03 | 每个画像维度保存值、来源、置信度、采集时间和版本。 | P0 | 任一组合决策可回溯到所用画像版本。 |
| UP-04 | 用户可查看、确认、修正或标记不确定的画像维度。 | P0 | 保存修正后生成新画像版本，不覆盖历史。 |
| UP-05 | 关键画像变更必须使关联授权书、目标组合和未执行订单失效。 | P0 | 失效对象显示受影响字段与重新评估入口。 |
| UP-06 | 系统不得用缺失信息或模型猜测自动提高风险边界。 | P0 | 缺失或低置信度只会收紧范围或要求补充信息。 |
| UP-07 | 系统将陈述风险、行为揭示风险与当前心智状态独立展示，并允许用户确认或纠正。 | P0 | 用户能看到差异、来源、置信度和对当前流程的影响。 |
| UP-08 | 行为、对话、日记和设备数据分别采用显式同意与最小化采集。 | P0 | 未同意的数据源不采集、不推断、不影响提示或订单。 |
| UP-09 | 高焦虑、冲动或显著偏差只能触发冷静期、附加确认、教育提示或复盘。 | P0 | 系统不能仅凭心智状态降低或提高目标权重、修改授权或创建订单。 |
| UP-10 | 系统必须基于财务全景与人生阶段生成可解释的财务健康摘要。 | P0 | 用户可看到现金缓冲、近期资金缺口、目标优先级和影响投资预算的主要因素。 |
| UP-11 | 面向用户的说明默认使用通俗语言，并提供数据、规则、来源和术语的展开层。 | P0 | 任何画像、组合、交易或风险事件都有简明解释与可展开的证据层。 |
| UP-12 | 用户可分别查看、启用、停用或撤回每类数据源的同意。 | P0 | 页面显示数据用途、隐私级别、最后采集时间；撤回后不再产生新的相关推断。 |

### 5.2 投资授权书

投资授权书是用户认知引擎到交易 Agent 的唯一可执行契约。它由用户确认，包含目标优先级、风险预算、资产范围、组合限制、交易自主级别、有效期和撤销状态。任何订单意图都必须引用一个有效授权书版本。

| 自主级别 | Agent 允许行为 | 用户控制 |
|---|---|---|
| L0 观察 | 研究、诊断、生成模拟组合。 | 不创建订单。 |
| L1 建议 | 生成目标组合与调仓计划。 | 用户逐次确认是否创建模拟订单。 |
| L2 仿真执行 | 在模拟账户中创建、执行和复盘订单。 | 用户可随时暂停策略或撤销未成交订单。 |
| L3 实盘预授权 | 在经审核的实盘账户中按约束执行订单。 | 用户选择逐笔确认、批次确认或预授权自动执行。 |

| ID | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| TM-01 | 系统从已确认画像生成投资授权书草案。 | P0 | 草案包含目标、风险预算、现金边界、资产范围和组合限额。 |
| TM-02 | 用户必须明确选择自主级别并确认后，授权书才生效。 | P0 | 默认级别为 L0；未确认不得创建订单意图。 |
| TM-03 | 授权书记录策略范围、再平衡频率、单笔限制、组合限制、有效期和撤销状态。 | P0 | 任一订单可查询对应授权字段。 |
| TM-04 | 用户可收紧、暂停或撤销授权；撤销立即阻止新订单。 | P0 | 已排队订单被取消或进入人工处理状态，并记录原因。 |
| TM-05 | L3 只能在实盘能力、账户状态和授权方式均已启用时出现。 | P0 | 未满足任一条件时，界面不展示可执行实盘入口。 |

### 5.3 Multi-Agent Workflow（多智能体协作系统）

#### 5.3.1 当前实现映射

当前代码没有按下表中的七个目标角色各实现一个同名 Agent。它使用统一 Agent 目录与场景工作流组合现有能力：

| 当前组件 | 已验证职责 | 尚未承担的目标职责 |
|---|---|---|
| `MultiAgentRuntime` | 将同步 `AgentRunner` 适配为异步调用；构造显式启用的模型与数据适配器 | 不管理用户、授权、组合、订单或长期工作流状态 |
| `Orchestrator` | 提供 `execute_multi_agent` 单一执行入口；缺少运行时时显式失败 | 不创建 TaskPlan，不自动降级为模拟结果 |
| Agent 目录 | 注册 43 个 Prompt、确定性 Monitor 和指标 Agent，并按能力、资源与权限路由 | 目录标签不是产品角色；交易域目标态增加 `financegod:planner` 后形成 44-Agent 能力池 |
| `WorkflowSelector` | 按用户暂停、硬风控、冷静期、数据可用性和授权状态选择工作流或阻断 | 当前只消费调用方传入的布尔状态，没有独立授权、风控或数据质量事实源 |
| `WorkflowExecutor` | 顺序执行阶段、阶段内并发 Agent、传递证据并生成最终产物 | 没有重试图、反馈轮次、持久化恢复或跨进程状态机 |
| 离线实验适配器 | 在不调用外部模型和市场数据的情况下验证选择、门禁、证据传递和产物结构 | 不能验证研究质量、实时数据、投资适当性、组合构造或交易执行 |

当前六个工作流定义和七个离线实验属于阶段 0 的技术基线，可复用于 P0，但不能替代下面的产品角色、数据对象和验收要求。

#### 5.3.2 MVP 目标角色

复杂投研问题不得由单一 Agent 独立完成。Multi-Agent Workflow 用 Planner 分解任务，以 Data、Research 和 Factor 提供可验证输入，再由 Strategy 形成候选、Backtest 验证、Report 解释。组合、风控和执行是工作流之外的受约束服务，只消费已通过工作流质量门的版本化产物。

| 角色 | 输入 | 输出产物 | 与其他角色的协作 | 禁止行为 |
|---|---|---|---|---|
| Planner Agent | 用户意图、有效授权、持仓、当前状态和工作流策略。 | TaskPlan、依赖关系、验收条件、优先级。 | 分派任务，接收失败和反馈，决定重试或终止。 | 自行研究、生成策略或提交订单。 |
| Data Agent | TaskPlan、批准的数据源和资产范围。 | DataPackage、市场环境、数据质量、时点和缺口。 | 为 Research、Factor、Backtest 提供同一版本数据。 | 以缺失或陈旧数据补全结论。 |
| Research Agent | DataPackage、研究任务和资产身份。 | ResearchMemo、事实、正反论据、风险、未知项。 | 向 Strategy 提供已引用的定性研究。 | 生成无证据的事实或投资指令。 |
| Factor Agent | DataPackage、资产池和因子任务。 | FactorStudy、因子定义、暴露、稳定性和适用范围。 | 向 Strategy 和 Backtest 提供可复算的量化研究。 | 将回测外推为未来收益承诺。 |
| Strategy Agent | 有效授权、ResearchMemo、FactorStudy、持仓和市场环境。 | StrategyProposal、假设、适用范围、风险情景和失效条件。 | 请求 Backtest 验证；根据失败反馈修订候选。 | 修改授权、绕过证据门槛或直接下单。 |
| Backtest Agent | StrategyProposal、版本化历史数据、费用和滑点假设。 | BacktestReport、指标、样本范围、局限、失效条件和通过状态。 | 向 Strategy 返回可操作的验证反馈；向 Report 提供限制。 | 使用未来数据、隐藏失败区间或批准订单。 |
| Report Agent | 已通过的研究、因子、策略和回测产物。 | UserReport、通俗解释、证据链接、不确定性和下一步。 | 面向用户说明结果；不参与策略改写。 | 将未验证结论包装为确定性建议。 |

#### 5.3.3 通信、反馈与持续优化

所有角色只能通过不可变的版本化产物通信。每个产物都包含工作流 ID、输入版本、来源、数据时点、模型或规则版本、置信度、状态、失败原因和责任角色。反馈只允许沿预定义路径回流：Backtest 可要求 Strategy 修订；Strategy 可要求 Research 或 Factor 补充；Planner 可基于失败重排任务。Report、组合、风控和执行不得反向修改研究或策略产物。

同一 TaskPlan 的策略修订必须有最大轮次和终止条件。达到上限、数据不足、回测未通过或授权失效时，工作流结束为失败或需要人工复核；不得静默回退到旧策略或默认结论。

| ID | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| AG-01 | Planner 必须将复杂请求拆为有输入、输出、依赖、优先级和验收条件的 TaskPlan。 | P0 | 任一工作流均可查看任务图与终止条件。 |
| AG-02 | Data Agent 只向工作流发布有来源、时点、适用范围、质量状态和缺口说明的 DataPackage。 | P0 | 陈旧、冲突或覆盖不足的数据不能进入 Research、Factor、Backtest 或执行输入。 |
| AG-03 | Research Agent 必须输出来源、事实、推断、正反观点、风险、未知项和失效条件。 | P0 | 缺少任一项时，研究状态为不可用于策略。 |
| AG-04 | Factor Agent 必须声明因子定义、计算版本、样本范围、暴露和稳定性。 | P0 | 无法复算或超出适用范围的因子研究不能进入策略。 |
| AG-05 | StrategyProposal 必须声明有效授权、研究与因子输入、市场假设、风险情景和失效条件。 | P0 | 未通过输入和字段校验的策略不能进入回测。 |
| AG-06 | Backtest Agent 必须使用版本化历史数据以及明确的费用、滑点和样本范围。 | P0 | BacktestReport 显示限制、失败区间和数据质量；通过状态不是未来收益承诺。 |
| AG-07 | Report Agent 仅消费已通过质量门的产物，并以通俗语言呈现结论与不确定性。 | P0 | 报告可链接全部证据和限制，不含未验证策略或保证收益。 |
| AG-08 | 每个 Agent 任务和产物记录工作流 ID、输入输出版本、时间、模型或规则版本、状态和责任角色。 | P0 | 审计页可追溯一次组合和订单的完整工作流链路。 |
| AG-09 | 反馈仅可沿 Planner 定义的路径回流，且有轮次上限、终止条件和显式失败状态。 | P0 | 回测失败只能反馈至 Strategy 或 Planner，不能直接触发组合或订单。 |
| AG-10 | 工作流优先级固定为：用户暂停与授权撤销、硬风控、心智冷静期、数据与市场异常、策略验证、定期维护。 | P0 | 低优先级任务不能覆盖高优先级阻断。 |

### 5.4 组合构造与调仓

组合构造服务在有效授权书、可用资产池、当前持仓和已批准策略的交集内工作。输出必须同时包括目标组合、约束满足报告、调仓计划、风险影响和无法满足的约束。不存在满足条件的组合时，服务必须返回不可构造结果。

| ID | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| PC-01 | 服务按授权书约束生成目标组合。 | P0 | 权重、资产范围、现金边界和集中度均通过规则校验。 |
| PC-02 | 服务比较当前与目标组合，生成最小必要调仓计划。 | P0 | 每笔交易意图说明其解决的偏离和对应策略依据。 |
| PC-03 | 组合生成考虑现金、最小交易单位、费用和模拟滑点。 | P0 | 无法满足时返回具体约束和受影响资产。 |
| PC-04 | 所有目标权重和调仓意图记录画像、授权、用户状态、策略、持仓、市场环境和数据版本。 | P0 | 同一输入可重放得到相同输出。 |
| PC-05 | 用户可查看组合风险、集中度、地区、行业、币种和情景影响。 | P0 | 界面同时显示数据覆盖率和计算时间。 |
| PC-06 | 未解析持仓超过阈值时，服务不得输出可执行调仓计划。 | P0 | 返回覆盖不足状态和需要修复的持仓行。 |

### 5.5 仿真交易与实盘接入契约

P0 的执行层是仿真账户。它使用真实市场数据或经声明的延迟数据，按可配置成交、费用和滑点模型处理订单。实盘与仿真使用同一订单意图和风控链路，但实盘必须额外完成账户、授权、市场与经纪商能力校验。

| ID | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| EX-01 | 仿真账户支持现金、持仓、订单、部分成交、拒单、取消、费用与滑点记录。 | P0 | 每次状态变化更新账户快照与审计事件。 |
| EX-02 | 每个订单意图在执行前通过授权、资产范围、现金、数量、价格保护和市场状态校验。 | P0 | 任一校验失败时订单不进入执行队列。 |
| EX-03 | 执行服务记录订单生命周期和执行质量。 | P0 | 用户可查看创建、批准、提交、成交、取消或拒绝的时间线。 |
| EX-04 | 仿真界面明确显示数据时点、成交假设、费用和滑点。 | P0 | 不得将仿真结果标为实际收益。 |
| EX-05 | 实盘适配器仅接受经授权的标准订单意图。 | P0 | 未启用实盘时适配器拒绝任何提交请求。 |
| EX-06 | 实盘模式必须支持立即暂停、未成交订单撤销和账户断连保护。 | P0 | 触发暂停后无新订单可被提交。 |

### 5.6 风控、熔断与人工介入

风险控制是独立于策略与执行 Agent 的强制服务。其规则分为不可覆盖的硬约束和可配置的软阈值。硬约束触发时订单必须拒绝；软阈值触发时订单进入暂停或人工复核状态，不得被 Agent 自动放行。

| 风控类别 | 典型规则 | 处置 |
|---|---|---|
| 用户授权 | 授权失效、级别不足、资产范围不符、集中度超限。 | 拒绝订单。 |
| 用户状态 | 用户主动暂停、冷静期未完成、心智状态低置信度或需附加确认。 | 暂停新订单并提示用户确认或复盘。 |
| 资金与订单 | 可用现金不足、数量异常、价格偏离、重复订单。 | 拒绝或暂停。 |
| 组合风险 | 目标偏离过大、单资产或行业超限、未解析持仓过高。 | 暂停调仓并要求复核。 |
| 市场与数据 | 市场关闭、交易暂停、报价陈旧、数据源异常。 | 停止提交，标记数据不可用。 |
| Agent 运行 | 模型版本未批准、任务超时、策略漂移、异常高频。 | 熔断策略并通知用户与运营方。 |

| ID | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| RK-01 | 风控服务在组合生成、订单创建和订单提交三个阶段运行。 | P0 | 同一订单有三段检查记录。 |
| RK-02 | 硬约束只能由新的用户授权或规则版本解除，不能由 Agent 覆盖。 | P0 | 拦截测试中 Agent 无法绕过拒绝结果。 |
| RK-03 | 用户可一键暂停所有策略和未来订单。 | P0 | 暂停操作在一个服务调用内生效并写入审计。 |
| RK-04 | 系统监控异常频率、重复订单、数据陈旧和策略漂移。 | P0 | 命中阈值后停止执行并创建可见风险事件。 |
| RK-05 | 所有阻断、暂停、熔断与人工复核都显示规则、输入、时间和下一步。 | P0 | 用户和运营方均可追溯事件。 |
| RK-06 | 触发冷静期时，系统暂停新订单并保留用户恢复路径。 | P0 | 只有用户确认或完成规定复盘后，且其余风险校验通过时才可恢复；不自动调整长期配置。 |

### 5.7 复盘、提醒与数据控制

| ID | 需求 | 优先级 | 验收标准 |
|---|---|---|---|
| RV-01 | 系统按月创建复盘任务，用户也可手动发起。 | P0 | 比较画像、授权、目标、实际持仓、策略和执行质量。 |
| RV-02 | 用户画像、授权书、市场数据或研究失效时，系统要求重新评估。 | P0 | 相关策略和未来订单显示失效状态。 |
| RV-03 | 用户可查看并导出画像、授权、组合、订单、成交和审计摘要。 | P0 | 导出包含版本、时间和关键假设。 |
| RV-04 | 用户可关闭非关键提醒，但不能关闭实盘授权撤销和严重风险通知。 | P0 | 提醒偏好与安全通知类别明确分开。 |
| RV-05 | 用户可发起删除申请；系统展示删除范围、保留要求和处理状态。 | P0 | 删除请求不会伪造或删除必要审计事实。 |
| RV-06 | 复盘提供画像变化、心智状态、市场环境、策略和执行质量的分层回顾。 | P0 | 用户能查看每类信号的使用范围，并修正错误推断。 |

## 6. 业务规则、数据与系统约束

### 6.1 不可违反的业务规则

| ID | 规则 |
|---|---|
| BR-01 | 不满足最低画像门槛时，不得生成个人化授权书或创建订单意图。 |
| BR-02 | 风险预算、资产范围、集中度和自主级别只能由已生效的授权书定义。 |
| BR-03 | 画像、授权、策略、组合、订单和成交必须使用可追溯版本关联。 |
| BR-04 | 交易 Agent 不得修改画像、授权或硬约束。 |
| BR-05 | 研究事实、模型推断、策略假设和风险事件必须明确区分。 |
| BR-06 | 缺少来源、反方观点、风险或失效条件的研究不得进入策略或组合构造。 |
| BR-07 | 数据缺失、资产身份无法确认、报价陈旧或市场状态异常时，必须显式失败或暂停。 |
| BR-08 | 未启用地区、资产、账户或经纪商不得产生实盘订单。 |
| BR-09 | 仿真账户、模拟成交和策略回测结果必须与实盘表现显式区分。 |
| BR-10 | 用户暂停、撤销授权或风险熔断后，系统不得接受新的执行订单。 |
| BR-11 | 用户可查看、修正和导出影响其个人化和交易结果的数据。 |
| BR-12 | 未解析资产不得被默认为现金、相近资产或宽基指数。 |
| BR-13 | 任何 Agent 失败不得静默降级为旧策略、空策略或看似成功的交易结果。 |
| BR-14 | 心智状态、行为与对话数据只能在用户单独同意的范围内采集和使用；撤回同意后不得用于新的推断或交易流程。 |
| BR-15 | 心智状态与市场情绪只能触发提示、冷静期、确认、复盘或暂停，不能直接修改长期授权、风险硬约束或创建订单。 |
| BR-16 | 目标组合、订单意图和用户报告只能消费通过 Multi-Agent Workflow 质量门的策略与回测产物。 |
| BR-17 | 角色反馈必须遵循 TaskPlan 中定义的路径、轮次和终止条件；任何角色不得绕过失败状态进入下游。 |

|---|---|---|

| UserStateSnapshot | 当前心智状态、认知偏差提示、信号来源、同意范围、置信度、有效期、用户确认状态。 | 解释、冷静期、确认和复盘输入。 |
| InvestmentMandate | 目标优先级、风险预算、现金边界、资产范围、限额、自主级别、有效期、状态。 | Agent 的唯一用户授权契约。 |
| HoldingSnapshot | 输入来源、持仓行、解析状态、未解析权重、估值时点、版本。 | 当前组合与风险计算。 |

| MarketContext | 市场、情绪与事件摘要、来源、数据时点、适用范围、置信度、风险事件、可用状态。 | 策略与执行前的市场环境输入。 |

| RiskEvent | 规则、严重度、输入快照、处置、操作者、恢复条件。 | 风控和熔断。 |
| AuditEvent | 事件类型、主体、时间、前后版本、请求关联 ID。 | 端到端追溯。 |

### 6.3 服务边界

| 服务 | 职责 | 不负责 |
|---|---|---|
| 画像服务 | 保存和计算量化画像、完整度、置信度和变更影响。 | 生成或提交交易订单。 |
| 用户状态服务 | 管理已同意的行为信号、状态快照、冷静期和用户确认。 | 修改授权、目标配置或创建订单。 |
| 授权服务 | 创建、签发、暂停、撤销和验证投资授权书。 | 替用户扩大授权范围。 |
| Multi-Agent Workflow 服务 | 执行 Planner、Data、Research、Factor、Strategy、Backtest、Report 的任务图、通信契约与反馈控制。 | 批准组合、覆盖风控或提交订单。 |
| 数据与市场环境服务 | 归集经批准数据源、发布 DataPackage、市场环境与数据质量状态。 | 在数据不足时生成交易结论。 |
| 回测服务 | 按版本化数据、费用、滑点和样本范围执行策略验证。 | 使用未来数据、隐藏失败区间或批准订单。 |
| 策略与组合服务 | 生成策略候选、目标组合和调仓计划。 | 绕过风险服务或直接执行订单。 |
| 风控服务 | 校验约束、监控事件、暂停和熔断。 | 被策略或执行服务覆盖。 |
| 执行服务 | 管理仿真订单与经批准的实盘适配器。 | 接受无有效授权的订单。 |
| 账户与经纪商适配器 | 处理授权账户状态、标准订单提交和回报。 | 存储经纪商密码或改变组合规则。 |
| 审计服务 | 固化版本、输入快照和事件链。 | 修改业务判断。 |

### 6.4 非功能需求

| 类别 | P0 要求 |
|---|---|
| 可解释性 | 每个画像、授权、组合和订单都能显示使用的版本、规则原因、数据时点和 Agent 输出。 |
| 决策可重放 | 给定相同画像、授权、数据、策略和规则版本，可重放组合与订单意图。 |
| 交易安全 | 订单幂等、状态机一致性、权限隔离、最小授权、暂停优先于执行。 |
| 数据质量 | 市场数据、主数据和研究证据显示来源、时点和可用状态；陈旧数据显式失败。 |
| 审计性 | 所有授权、风控、订单和人工动作都保留不可歧义的事件链。 |
| 协作可追溯性 | 所有任务图、角色通信、反馈轮次、产物版本和终止原因均可查询与重放。 |
| 性能 | 建档、画像和组合预览提供交互式反馈；研究和订单执行使用异步进度。 |
| 隐私 | 最小化采集、加密存储、访问审计；不保存经纪商密码。 |
| 敏感状态治理 | 心智、行为和对话信号具有独立同意、用途限制、有效期和撤回路径。 |
| 易理解性 | 默认解释必须使用通俗语言并说明下一步；专业数据、术语、规则和证据按需展开。 |
| 失败可见性 | 所有 Agent、数据、组合和执行失败均返回原因、影响范围和可执行下一步。 |

| POST /v1/reviews | 创建或完成复盘。 | 不生成未授权的交易动作。 |

## 8. 验收测试与上线门槛

### 8.1 核心验收场景

| 场景 | 前置条件 | 操作 | 预期结果 |
|---|---|---|---|
| A1 画像成功 | 用户填写所有关键资料。 | 确认画像。 | 先生成财务健康摘要，再生成带来源、完整度、置信度和版本的画像。 |
| A2 画像冲突 | 风险偏好高但现金需求紧张。 | 生成授权书草案。 | 系统解释冲突，并采用审慎风险边界。 |
| A3 授权不足 | 用户停留在 L0。 | 尝试创建模拟订单。 | 系统拒绝，说明需要启用对应自主级别。 |
| A4 组合构造成功 | 有有效授权、支持资产和完整持仓。 | 生成目标组合。 | 返回权重、约束报告、风险解释和调仓计划。 |
| A5 持仓覆盖不足 | 未解析权重超过阈值。 | 生成调仓计划。 | 返回不可构造，列出需要修复的持仓。 |
| A6 研究不足 | 缺少反方或关键证据。 | 发起策略生成。 | 研究不可用于策略，任务不继续。 |
| A6a 回测未通过 | 策略候选通过研究与因子输入校验，但验证结果未通过。 | 完成 Backtest。 | 仅向 Strategy 或 Planner 返回反馈；不得生成目标组合、订单或正向用户报告。 |
| A6b 反馈上限 | 同一策略修订达到最大反馈轮次。 | Planner 继续请求修订。 | 工作流以需要人工复核或失败结束，不回退旧策略。 |
| A7 仿真执行成功 | L2 授权、风险校验通过、市场可用。 | 执行调仓计划。 | 订单经历完整状态机，账户、费用和滑点同步更新。 |
| A8 风控阻断 | 订单将突破单资产或现金边界。 | 创建订单意图。 | 订单被拒绝，不进入执行队列。 |
| A9 用户暂停 | 有排队的仿真订单。 | 点击暂停所有策略。 | 新订单停止，未成交订单取消或转人工处理。 |
| A10 授权失效 | 用户修改期限或撤销授权。 | 查看策略和订单。 | 关联策略、组合和订单均标为失效。 |
| A11 实盘未启用 | 未配置账户或经纪商。 | 尝试实盘提交。 | 接口和界面均拒绝，不能降级为假成功。 |
| A12 心智冷静期 | 用户确认高焦虑状态或触发已同意的冷静期规则。 | 尝试创建新订单。 | 新订单暂停，显示原因和恢复条件；目标配置和授权保持不变。 |
| A13 市场环境不可用 | 数据陈旧、来源冲突或市场状态异常。 | 生成策略或提交订单。 | 服务显式失败或暂停，不能以旧环境或默认结论继续。 |
| A14 顾问式解释 | 用户查看财务健康、画像或组合建议。 | 阅读默认说明并展开详情。 | 默认内容以通俗语言说明现状、影响和下一步；展开层显示数据、规则、来源和术语。 |

### 8.2 上线阻断项

- 任一路径在没有有效画像和授权书时生成可执行订单；
- 任一 Agent 可以直接修改用户约束、跳过风控或提交订单；
- 已暂停、已撤销或已过期授权下仍能创建新订单；
- 研究证据不足、持仓覆盖不足或数据陈旧时仍产生正向交易结论；
- 未经过 Backtest 验证、反馈上限或质量门的策略进入组合、订单或用户报告；
- 仿真订单结果被展示为实盘业绩或保证收益；
- 订单、成交或风险事件无法关联到用户、授权、组合、规则和数据版本；
- 心智状态、浏览行为或对话情绪在缺少同意时被采集，或被直接用来修改授权和创建订单；
- 用户无法暂停 Agent、撤销授权、查看订单原因或导出关键记录。

## 9. 依赖、风险与待决策项

### 9.1 外部依赖

| 依赖 | 影响 | 所有者 |
|---|---|---|
| 证券与基金主数据 | 资产身份、交易属性、资产池和持仓分类。 | 数据团队 |
| 市场数据与数据许可 | 研究时效、仿真成交、风险与执行质量。 | 数据与法务 |
| 经纪商与账户授权 | P1 实盘账户状态、订单提交和成交回报。 | 平台、安全与业务 |
| LLM 与受控检索 | 研究整理、策略解释和画像交互。 | AI 与平台 |
| 优化器与规则服务 | 组合构造、风险约束和重放能力。 | 量化与平台 |
| 审计与事件基础设施 | 订单状态、风控、人工介入和问题排查。 | 平台团队 |

### 9.2 主要风险与缓解措施

| 风险 | 缓解措施 |
|---|---|
| 画像错误导致不匹配策略 | 显示来源与置信度，要求用户确认关键维度，关键变更使授权失效。 |
| 心智状态误判或用户感到被监控 | 单独同意、最小化采集、置信度与解释、用户校正、冷静期而非自动交易。 |
| Agent 越权交易 | 授权书是唯一契约，独立风控三段校验，暂停和撤销优先。 |
| 多智能体协作失控 | 用 TaskPlan、版本化产物、预定义反馈路径、轮次上限和终止状态约束通信；全链路记录责任角色。 |
| 模型编造或旧数据驱动策略 | 研究证据门槛、数据时点、显式失败和策略有效期。 |
| 仿真与实盘差异被忽略 | 明示成交、费用和滑点假设；实盘适配器独立评审。 |
| 订单重复或状态错乱 | 标准订单意图、幂等键、状态机和审计事件。 |
| 用户过度依赖自动化 | 逐级自主授权、组合解释、风险事件和一键暂停。 |
| 市场、数据或合规范围不匹配 | 按地区、账户和资产池启用能力，未启用时拒绝执行。 |

### 9.3 上线前待决策项

1. P0 首发市场、用户地区、资产池和市场数据供应商；
2. P0 的单市场范围与后续跨市场扩展顺序；
3. 量化画像与心智状态的最小问题集、同意范围、置信度计算方法和更新频率；
4. 冷静期、附加确认和认知偏差提示的触发阈值、恢复条件与运营责任；
5. 授权书的风险预算、集中度、未解析持仓和价格保护阈值；
6. P0 仿真成交、费用和滑点模型，以及数据延迟披露方式；
7. P1 实盘的首个经纪商、授权模式、账户类型和用户身份流程；
8. L3 实盘预授权的最大范围、人工复核条件和严重事件处置流程；
9. 各类审计记录、用户状态、订单记录的保留、导出与删除策略。

### 9.4 建议交付阶段

| 阶段 | 当前状态 | 目标 | 交付范围 | 退出条件 |
|---|---|---|---|---|
| 阶段 0：运行时与工作流基线 | 部分完成 | 建立可复用、可审计且明确失败的 Multi-Agent 技术底座。 | 固定版本运行时、43-Agent 基础目录、统一请求/结果契约、6 个工作流定义、7 个离线场景、证据传递和结构化产物。 | 清理遗留工作流契约；全量单元测试通过；安装、示例入口和运行说明可从干净环境复现。 |
| 阶段 A：理解与授权 | 未开始 | 验证用户是否愿意建立和修正画像。 | 单市场、画像、心智状态的显式同意、授权书、持仓诊断、L0 观察。 | 用户能理解并确认画像与授权；无未经同意的状态采集。 |
| 阶段 B：工作流与仿真 | 未开始 | 验证 Multi-Agent Workflow 能否在约束下完成一轮交易闭环。 | 将现有运行时映射到 Planner、Data、Research、Factor、Strategy、Backtest、Report，并交付组合、风控、冷静期和 L2 仿真订单。 | 全部仿真订单可追溯；未通过质量门的策略无法进入组合和订单。 |
| 阶段 C：受控实盘试点 | 未开始 | 验证受授权执行的账户、运行和应急机制。 | 单一经纪商、受限资产池、逐笔或批次确认、断连保护和人工处置。 | 账户、授权、订单和风险事件通过独立安全与合规评审。 |

## 10. 术语

| 术语 | 定义 |
|---|---|
| 量化用户画像 | 对目标、约束、风险、行为和偏好的版本化结构化表达，包含来源和置信度。 |
| 用户状态快照 | 对当前心智状态与认知偏差信号的、经同意且有有效期的表达，不等同于长期风险授权。 |
| 市场环境 | 对市场情绪、事件、数据质量和适用范围的版本化描述，不构成交易指令。 |
| Multi-Agent Workflow | 由 Planner、Data、Research、Factor、Strategy、Backtest 与 Report 通过版本化产物协作完成的投研与验证流程。 |
| TaskPlan | Planner 创建的任务图，定义角色、依赖、验收条件、反馈路径、轮次上限和终止条件。 |
| BacktestReport | 对策略候选在版本化历史数据和明确假设下的验证报告，不代表未来收益。 |
| 投资授权书 | 用户确认的、约束 Agent 行为的可执行契约。 |
| 自主级别 | 用户授予 Agent 从观察、建议、仿真执行到实盘预授权的权限范围。 |
| 策略候选 | 在指定授权和市场假设下可供组合构造服务评估的策略方案。 |
| 目标组合 | 满足授权书和风险约束的资产权重集合。 |
| 订单意图 | 经组合服务生成、尚未提交执行的标准化交易请求。 |
| 仿真交易 | 使用声明的市场数据、成交、费用和滑点模型处理的非实盘交易。 |
| 风险熔断 | 因硬约束、异常运行或风险事件而停止策略和未来订单的强制状态。 |
| 冷静期 | 因用户主动确认或已同意的状态规则而暂停新订单、要求额外确认或复盘的临时状态。 |
| 执行质量 | 订单在成交、费用、滑点、延迟和拒绝等维度的实际或模拟表现。 |
