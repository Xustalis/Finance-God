## 1. 文档目的与产品定义

### 1.1 文档目的

本文定义 Finance-God 交易域的 MVP 目标态：以 44-Agent 能力池和确定性交易服务为基础，完成从市场与标的研究、策略验证、组合构造、交易计划、订单复核、仿真执行到交易复盘的可解释、可追溯闭环。

本文是产品需求与验收依据，不是用户画像设计文档，也不描述当前实现进度。用户画像、心智状态建模和投资授权书的生成由上游模块负责；交易域只读取已经确认的授权、资产范围、限额、冷静期和撤销状态。

### 1.2 一句话定位

**Finance-God 交易台是一套由多智能体完成研究和建议、由确定性服务掌握资金与订单事实、由用户保留最终交易控制权的跨市场仿真交易工作台。**

### 1.3 首要产品目标

建立一条可验证的交易闭环：

```mermaid
flowchart LR
    A["已确认的授权与状态输入"] --> B["Planner 选择或创建工作流"]
    B --> C["研究、市场、组合与策略工作流"]
    C --> D["质量门与治理结论"]
    D --> E["组合构造与交易计划"]
    E --> F["订单草稿与最终复核"]
    F --> G["提交前确定性风控"]
    G --> H["仿真执行与订单事实"]
    H --> I["组合更新、复盘与策略监控"]
```

首要成功标准为：

- 用户能够从任一计划或订单回溯到使用的数据、工作流、Agent、策略、规则和确认记录；
- Agent 失败、数据不足、低置信度和阻断均显式呈现，不生成默认结论或伪造成功；
- Agent 不得修改资金、持仓、订单、成交、授权或硬风控事实；
- 仿真结果、Agent 建议和回测均不构成收益保证或真实成交。

### 1.4 产品原则

| 原则 | 产品要求 |
|---|---|
| 事实与建议分离 | 数据、规则、订单和成交事实与 Agent 推断、建议使用不同状态与视觉层级。 |
| 用户保留控制权 | 自动工作流可以生成待审阅计划，但订单必须经过用户最终确认。 |
| 确定性服务掌权 | 组合优化、资金账本、正式风控、撮合和状态转换由确定性服务执行。 |
| 失败显式可见 | 不静默重试到成功，不回退旧结论，不用空结果冒充正常结果。 |
| 证据优先 | 事实、推断和建议必须区分，并记录来源、时点、范围、置信度、未知项与失效条件。 |
| 最小必要调用 | 44 个 Agent 构成能力池；单次任务只运行必要 Agent，不为凑数量调用。 |

## 2. 范围、角色与使用边界

### 2.1 MVP 范围

以下能力全部属于 MVP/P0：

1. 上游授权状态与冷静期结果的只读消费；
2. 人民币单一仿真账户、初始现金设置与空仓起步；
3. 自选、Agent 候选与跨市场标的搜索；
4. 44-Agent 能力目录、15 条正式工作流和动态 TaskPlan；
5. 大A、港股、美股、ETF、LOF和场外公募基金研究；
6. 日线行情、技术指标、研究图层和上下文对话；
7. 策略开发、回测、稳定性验证与隔离代码执行；
8. 组合分析、目标权重、确定性优化和再平衡；
9. 单标的与多标的交易计划；
10. 订单草稿、手动订单轻量复核和最终确认；
11. 提交前单次正式风控；
12. 跨市场日线仿真撮合、港美股卖空和场外基金申赎；
13. 订单、成交、撤单、公司行动、复盘、审计和站内通知。

### 2.2 支持市场与资产

| 市场/资产 | 研究 | 仿真交易 | P0 说明 |
|---|---:|---:|---|
| 大A股票 | 是 | 是 | 仅做多；按交易日历、涨跌停、最小单位、费用和结算规则仿真。 |
| 港股 | 是 | 是 | 支持做多与卖空；模拟可借数量、保证金、借券费、召回和强平。 |
| 美股 | 是 | 是 | 支持做多与卖空；模拟市场时区、费用、公司行动和卖空限制。 |
| ETF、LOF | 是 | 是 | 使用交易所订单模型；研究包含指数、持仓穿透、溢折价、跟踪与流动性。 |
| 场外公募基金 | 是 | 是 | 使用申购、赎回、转换和定投模型；处理截止时间、未知净值和确认日。 |
| 指数、宏观、行业 | 是 | 否 | 作为研究、基准、市场环境和压力测试输入。 |
| 期货、期权 | 是 | 否 | 只作为相关性、波动率、情绪与风险研究数据。 |

### 2.3 唯一产品用户角色

本 PRD 只定义“终端投资用户”。该角色可以查看研究、管理自选、设置仿真账户初始现金、审阅计划、修改草稿、确认订单、撤单、跟踪执行和完成复盘。

本文不设计用户画像，也不定义运营、客服、风控、合规或系统管理员的产品界面。

### 2.4 核心用户故事

- 作为投资用户，我希望系统针对当前市场、标的和组合选择合适的 Agent 工作流，而不是给出单模型结论。
- 作为投资用户，我希望看到支持意见、反方意见、未知项和失效条件，再决定是否采用计划。
- 作为投资用户，我希望修改 Agent 建议的数量、价格或风险参数，并立即看到费用、组合影响和策略偏离。
- 作为投资用户，我希望在 Agent 暂停或不可用时仍能提交满足确定性规则的手动订单。
- 作为投资用户，我希望冷静期只阻止新增风险，不妨碍降低风险的卖出或赎回。
- 作为投资用户，我希望每个订单状态都来自仿真执行事实，而不是由 Agent 推测。
- 作为投资用户，我希望复盘研究假设、计划、修改、成交与组合结果之间的差异。

### 2.5 明确不包含

- 用户画像、心智状态模型和投资授权书的创建或修改；
- 实盘交易、经纪商接入、资金划转和真实账户授权；
- 期货、期权、加密资产或其他衍生品下单；
- 移动端页面、管理后台、文件导出、邮件和移动推送；
- Tick 数据和高频、超短线或盘中自动交易；
- 已有持仓手动录入或 CSV 导入；
- 由 Agent 覆盖硬风控、修改账本或伪造成交；
- 收益承诺、确定性价格预测或将仿真表现标为真实业绩。

## 3. 信息架构与端到端旅程

### 3.1 能力域与页面映射

PRD 按业务能力域组织，页面设计沿用 T00–T10 的信息架构；M01 移动端不在范围内。

| 能力域 | 页面映射 | 核心任务 |
|---|---|---|
| 授权状态与账户 | T00 | 查看上游授权、暂停 Agent、设置仿真初始现金。 |
| 总览与待办 | T01 | 查看风险、工作流、计划、订单、复盘和通知待办。 |
| 自选与候选 | T02 | 管理自选分组，查看 Agent 候选、比较和解释。 |
| 研究交易台 | T03 | 研究标的、运行工作流、对话、生成和修改订单草稿。 |
| 交易计划 | T04 | 查看单标的或多标的计划、组合影响、分歧和版本。 |
| 订单草稿 | T05（T03 内部） | 编辑方向、类型、数量、价格、有效期和风险参数。 |
| 最终复核 | T06（T03 覆盖层） | 查看不可编辑摘要、正式风控结果并确认提交。 |
| 执行中心 | T07 | 跟踪订单、部分成交、撤单、拒单、异常和批次。 |
| 组合与偏离 | T08 | 查看当前/目标/计划后组合、风险与偏离。 |
| 交易复盘 | T09 | 对比假设、计划、修改、执行和组合结果。 |
| 过程与证据 | T10 | 查看工作流图、Agent 贡献、证据、版本和错误。 |

### 3.2 主旅程

1. 用户进入 T00，查看有效授权与冷静期结果，设置人民币仿真账户初始现金。
2. 系统在各市场收盘数据可用后更新日线数据，并按固定日程运行市场、策略与监控工作流；重大事件触发专项工作流。
3. 用户从 T02 自选或候选进入 T03，Planner 选择正式工作流或在固定安全约束下创建 TaskPlan。
4. 工作流生成研究、市场、组合或策略产物；只有 `completed` 且通过确定性质量门的产物能进入自动交易计划。
5. 自动工作流可以生成待用户审阅的计划草案，但不能创建订单。
6. 用户在 T04 审阅计划，在 T03/T05 创建或修改订单草稿。
7. T06 展示不可编辑摘要并在提交前生成唯一正式 `RiskCheckResult`；硬阻断时不提供提交操作。
8. 用户确认后，确定性仿真服务受理订单；Agent 只监控和解释，不决定成交。
9. 成交事实更新账户和组合；T09 生成复盘，策略监控工作流跟踪失效条件与漂移。

### 3.3 手动订单旅程

手动订单可以绕过研究和策略工作流，但必须使用轻量 `order_review(mode=manual)`：

- 输出费用、组合影响、策略偏离和风险解释；
- Agent 失败不阻止通过确定性规则的合法手动订单；
- 冷静期内只允许降低风险的卖出或赎回；
- 用户暂停 Agent 后不调用该 Agent 工作流，但仍可经过提交前风控完成合法手动订单；
- 硬风控绝不可覆盖，软风险可在查看原因并二次确认后继续。

## 4. Multi-Agent 总体架构

### 4.1 架构组成

MVP 使用 44-Agent 能力池：43 个基础目录 Agent，加一个产品级 Planner Agent `financegod:planner`。

| 层级 | 职责 | 权限边界 |
|---|---|---|
| Planner | 理解任务、选择正式工作流或创建动态 TaskPlan、安排依赖与并发、汇总状态。 | 不研究、不决定交易、不修改授权、账本或订单。 |
| Prompt Agent | 研究、辩论、解释、形成条件化建议或开发受限策略代码。 | 只能使用提供的证据；不得声称已执行外部动作。 |
| 确定性 Monitor | 基于批准数据计算市场、拥挤、相关性和波动状态。 | 输出研究证据，不输出交易事实。 |
| 确定性产品服务 | 数据质量、组合优化、资金账本、正式风控、撮合、订单状态和公司行动。 | 掌握事实；不让 Agent 覆盖规则。 |
| 审计存储 | 保存运行、路由、输入版本、错误、阻断和事实事件。 | 记录不可被 Agent 修改或删除。 |

### 4.2 Agent 与服务的不可越权规则

1. Agent 的 `proposed_actions` 始终是待审核建议。
2. Portfolio、Risk 或 Trader 类 Agent 可以提供解释与候选，但最终权重由确定性优化器求解。
3. 只有执行服务能创建 `Order`、`Fill` 和账户账本事件。
4. 最后一个治理 Agent决定“建议通过/建议不通过”；系统仍执行字段、证据、数据新鲜度和硬风险校验。
5. 任何 Agent 都不能覆盖用户暂停、硬风控、冷静期限制或授权撤销。
6. SSQuant、Panda Trading 等带交易能力标签的 Agent 只能在隔离仿真环境生成建议、代码或仿真请求，禁止实盘外部动作。

### 4.3 调用类型

| 调用类型 | 定义 |
|---|---|
| 必选 | 对应工作流变体缺少该 Agent 或同等确定性服务时不能满足质量门。 |
| 条件调用 | 仅在市场、资产、数据、任务或证据条件满足时调用。 |
| 禁止调用 | 当前场景不具备资源、权限或业务合理性时不得调用。 |

## 5. 工作流编排与治理

### 5.1 正式工作流与动态 TaskPlan

系统提供 15 条版本化正式工作流。Planner 可以直接选择正式工作流，也可以在运行时创建动态 TaskPlan，无需逐次人工批准，但必须遵守以下系统约束：

- Agent、工具、数据源、服务节点和外部动作白名单；
- 用户暂停、硬风控、冷静期、数据状态和授权优先级；
- 节点级与工作流级超时、按错误类型配置的重试以及总次数/总时长上限；
- 策略代码的隔离文件、网络和外部动作权限；
- 所有资金、订单和成交写入必须通过确定性服务。

动态工作流定义只要求保存 Agent 列表与最终产物。系统审计独立保存运行 ID、输入版本、路由、执行顺序、状态、错误、权限和阻断，以满足可追溯要求。

### 5.2 执行模型

- 同一阶段中无依赖 Agent 可以并行；
- 阶段之间按依赖顺序执行；
- 单个 Agent 失败使当前阶段失败，并按错误类型重试；
- 达到节点或工作流总上限后显式失败，不自动换成旧结果；
- 已被确定性执行服务受理的订单不因监控或解释 Agent 失败而撤销；
- 每个节点必须标记为 `required` 或 `non_blocking`；手动订单解释 Agent 和已受理订单监控 Agent属于 `non_blocking`；
- 自动交易链路中，Planner 选择的依赖集合在计划创建时固定；其中每个工作流必须为 `completed` 且 `trade_eligible=true` 才能进入下一阶段。

### 5.3 状态

| 状态 | 定义 | 是否可进入自动交易计划 |
|---|---|---:|
| `queued` | 等待资源或前序依赖。 | 否 |
| `running` | 正在执行。 | 否 |
| `completed` | 全部必需阶段完成且质量门通过。 | 是 |
| `attention_required` | 低置信度、证据不足、冷静期或数据问题需要处理。 | 否 |
| `failed` | 超时、错误或质量门失败。 | 否 |
| `timed_out` | 达到节点或工作流总时长上限。 | 否 |
| `blocked` | 用户暂停或硬风控直接终止。 | 否 |
| `expired` | 依赖数据、授权、持仓、规则或产物版本变化。 | 否 |

`completed` 只表示工作流执行完成；可否进入交易下游由独立字段 `trade_eligible` 决定。只读研究可以是 `completed`，但必须设置 `trade_eligible=false`。

### 5.4 选择与阻断优先级

优先级固定为：

1. 用户暂停；
2. 硬风控；
3. 冷静期；
4. 数据与市场异常；
5. 授权状态；
6. 正常研究、策略、计划与执行工作流。

用户暂停与硬风控不调用 Agent，直接生成 `WorkflowBlockNotice`。用户暂停只停止新 Agent 工作流，不阻止满足确定性规则的手动订单；硬风控阻止对应订单和新增风险行为。

### 5.5 数据不足与低置信度

- 低置信度或证据不足的产物标记为 `attention_required`；
- 数据陈旧、冲突或缺失时触发 `data_quality_review`；
- `company_research` 可以使用公告、财务和已有证据继续只读研究，但必须标记行情缺失，且产物不能进入交易计划；
- 不允许以模型猜测、旧策略或最近一次成功结果补齐缺失数据。

## 6. 十五条正式工作流

### 6.1 工作流总表

| ID | 工作流 | 触发条件 | 核心阶段 | 最终产物 | 用户可见结果 |
|---|---|---|---|---|---|
| WF-CR-01 | `company_research` | 用户研究大A、港股或美股公司，或上游计划需要公司证据。 | 事实审阅 → 正反观点与风险辩论 → 研究治理汇总 | `ResearchMemo` | 公司事实、投资论点、反方证据、风险、未知项和治理结论。 |
| WF-MC-01 | `market_context` | 定时市场更新、重大市场变化或交易计划需要环境判断。 | 市场监控 → 新闻/情绪解释 → 治理复核 | `MarketContext` | 市场状态、驱动因素、数据限制、风险环境和适用范围。 |
| WF-PS-01 | `portfolio_stress` | 组合集中度、相关性、拥挤或波动异常。 | 相关性监控 → 拥挤风险监控 → 风险辩论与治理 | `PortfolioRiskReview` | 压力情景、脆弱点、分歧和限制。 |
| WF-SV-01 | `strategy_validation` | 新策略、参数修改、策略过期或重新验证。 | 需求 → Alpha 设计 → 回测 → 验收测试 → 治理汇总 | `StrategyValidationDossier` | 回测指标、样本内外结果、成本、稳定性、失效条件和通过建议。 |
| WF-RO-01 | `review_only` | 冷静期生效或策略授权无效。 | 证据与下行风险只读复核 | `ReviewOnlyMemo` | `attention_required`；只读证据、下行风险和可执行下一步。 |

### 6.2 共同输入与输出规则

所有正式工作流必须接收：

- 唯一 `workflow_id`、工作流版本和请求意图；
- 当前市场、资产身份和数据包版本；
- 有效授权、冷静期、用户暂停和硬风控状态；
- 需要时接收当前持仓、账户、策略、计划或订单版本；
- 可用资源、允许动作和 Agent/服务节点权限。

所有工作流产物必须包含：

- 产物类型、状态、工作流 ID、生成时间和适用范围；
- `trade_eligible` 及其确定性质量门理由；
- 输入版本、参与 Agent/服务节点、路由原因和失败节点；
- 事实、推断、建议、反方意见、未知项和失效条件；
- 证据 ID、数据时点、模型/规则版本和置信度；
- 待审核动作，且明确标注“未执行”。

### 6.3 正常研究工作流

#### 6.3.1 `company_research`

| 项目 | 要求 |
|---|---|
| 资产范围 | 仅公司型大A、港股和美股标的；ETF、LOF和场外基金改用 `fund_research`。 |
| 必需输入 | 公司身份、公告/财务/行情/新闻证据、市场、研究时点。 |
| 质量门 | 公司身份无冲突；事实均有来源；至少包含支持、反方、风险、未知项和失效条件；治理 Agent 给出通过建议。 |
| 数据异常 | 市场行情不可用时允许只读研究，但产物不能进入计划。 |
| 失败路径 | 证据不足或治理不通过时为 `attention_required`；节点重试耗尽时为 `failed`。 |

#### 6.3.2 `market_context`

| 项目 | 要求 |
|---|---|
| 调度 | 各市场收盘数据可用后运行；重大市场事件可额外触发。 |
| 必需输入 | 日线行情、资金/事件/波动数据、新闻和宏观证据。 |
| 质量门 | 数据覆盖和时间对齐通过；市场状态与解释分离；结论声明适用市场和期限。 |
| 失败路径 | 数据不可用时切换 `data_quality_review`，不得生成正常市场结论。 |

#### 6.3.3 `portfolio_stress`

| 项目 | 要求 |
|---|---|
| 必需输入 | 当前组合版本、持仓穿透、市场数据、相关性、融资/拥挤和波动证据。 |
| 确定性节点 | 相关性、集中度、重叠、损失情景和资金压力由确定性计算器生成。 |
| 质量门 | 所有持仓身份可解析；场景参数与数据时点完整；风险辩论包含上行、基准和下行观点。 |
| 失败路径 | 组合覆盖不足或数据冲突时触发 `data_quality_review`。 |

#### 6.3.4 `fund_research`

| 项目 | 要求 |
|---|---|
| ETF/LOF | 指数身份、持仓穿透、跟踪误差、溢折价、流动性、费用、风格暴露和替代品。 |
| 场外基金 | 净值、基金经理、持仓、风格漂移、回撤、费用、申赎/转换/定投规则和同类比较。 |
| 质量门 | 份额类别、币种、交易/申赎方式和费用版本明确；不可将公司估值模板强套于基金。 |
| 失败路径 | 基金接口标记为废弃、未上线或未通过接入验收时，返回数据不足，不生成替代值。 |

### 6.4 策略与组合工作流

#### 6.4.1 `strategy_validation`

最低回测指标包括收益、年化波动、最大回撤、Sharpe、Sortino、Calmar、胜率、换手率、交易成本、样本内/样本外和稳定性。费用、滑点和市场规则使用版本化假设。

策略代码只能在隔离工作区执行，限制文件、网络和外部动作；测试通过后才可用于回测。未来数据、隐藏失败区间、缺少样本外验证或无法复算时不得通过质量门。

#### 6.4.2 `portfolio_construction`

Agent 生成候选权重和解释，确定性优化器在授权、现金、集中度、卖空、交易单位、费用和市场规则内求解。输出至少覆盖权重、现金、地区、行业、币种、因子、集中度、相关性、基金重叠、波动、回撤和压力情景。

不存在满足约束的组合时，必须返回“不可构造”以及冲突约束，不得放宽硬限制。

#### 6.4.3 `trade_plan_generation`

计划可以是单标的、多标的、再平衡或观察计划。自动工作流只生成“待用户审阅”的计划草案。每个计划必须展示使用的工作流、产物版本、失效时间、未解决分歧、交易目的、不行动影响、预计费用和计划后组合。

### 6.5 订单与执行工作流

#### 6.5.1 `order_review`

计划订单与手动订单均进入订单复核。计划订单必须引用有效 `TradePlan`；手动订单使用 `order_review(mode=manual)` 轻量模式，不新增第 16 条正式工作流。该工作流完成草稿计算、Agent 解释、提交前唯一正式风控和不可编辑摘要。Agent 解释失败不应阻止通过确定性规则的手动订单，但失败必须显式显示。

#### 6.5.2 `simulation_execution`

该工作流包含确定性服务节点。正式订单事实只能由仿真执行服务产生：

1. `order_review` 已生成未过期的正式 `RiskCheckResult`；
2. 用户查看不可编辑摘要，处理可覆盖的软风险并明确确认；
3. 执行服务以幂等键受理请求；
4. 市场规则与撮合模型生成 `Order`、`Fill` 和账本事件；
5. Agent 监控和解释执行结果，但失败不回滚已经受理的事实。

#### 6.5.3 `post_trade_review`

复盘对比研究假设、计划、用户修改、正式风控、成交、费用、滑点、组合影响和失效条件，并生成下一步观察项。复盘结论可以触发 `strategy_monitoring`，但不能自动修改策略或创建订单。

### 6.6 专项与保护工作流

#### 6.6.1 `review_only`

冷静期或授权失效时只复核证据和下行风险，固定返回 `attention_required`。冷静期内只允许降低风险的卖出或赎回。

#### 6.6.2 `data_quality_review`

诊断空数据、字段缺失、标的冲突、时间错位、来源冲突和接口不可用。输出受影响工作流、资产和产物版本，以及明确修复动作。

#### 6.6.3 `event_impact`

事件事实与影响推断必须分离。事件工作流不能因为“重大”而覆盖数据质量、授权、冷静期或正式风控。

#### 6.6.4 `cross_market_analysis`

跨市场计算使用内部原币、人民币折算汇率、来源和时点；界面只显示人民币结果。市场交易日和时区不一致时必须标记时间对齐限制。

#### 6.6.5 `strategy_monitoring`

比较当前输入、策略批准版本和历史基线。命中失效条件时将策略标记为 `expired` 或建议暂停，不自动切换旧策略。

### 6.7 无 Agent 阻断路径

| 阻断 | 结果 | 下一步 |
|---|---|---|
| 用户暂停 | 不调用 Agent，生成 `WorkflowBlockNotice`；不影响合法手动订单。 | 用户明确恢复后才可创建新 Agent 工作流。 |
| 硬风控 | 不调用 Agent，不生成策略、计划或订单意图。 | 处理硬风险或等待有效授权/规则状态变化。 |

## 7. 44-Agent 能力目录

### 7.1 映射规则

工作流简称：`CR` 公司研究、`MC` 市场环境、`PS` 组合压力、`SV` 策略验证、`RO` 只读复核、`DQ` 数据质量、`FR` 基金研究、`PC` 组合构建、`TP` 交易计划、`OR` 订单复核、`SE` 仿真执行、`PR` 交易后复盘、`EI` 事件影响、`CM` 跨市场分析、`SM` 策略监控。

- `M` 表示工作流启动且资产、数据和资源条件满足时必选；
- `C` 表示满足括号中的业务条件时调用；
- 未列入 `M/C` 的工作流默认禁止调用；
- `M` 不覆盖资源门禁；资源缺失时必须失败或进入数据质量诊断。

所有 Agent 共同禁止：

- 创建或修改订单、资金、持仓、成交、授权或正式风控事实；
- 覆盖用户暂停、硬风控或冷静期；
- 把建议、代码、请求或模拟结果表述为已执行动作；
- 脱离输入证据生成事实、目标价、仓位、适当性结论或对外发布内容。

## 8. 数据能力与质量治理

### 8.1 PandaData 目标范围

PandaData 是主要结构化数据源。生产能力目录当前明确开放以下 26 个端点：

- A 股、港股、美股与指数：`get_stock_rt_daily`、`get_stock_daily`、
  `get_stock_min`、`get_stock_rt_min`、`get_hk_daily`、`get_us_daily`、
  `get_hk_detail`、`get_us_detail`、`get_index_daily`、`get_index_detail`、
  `get_index_weights`；
- 期货与期权研究：`get_future_daily`、`get_future_detail`、
  `get_future_dominant`、`get_option_detail`、
  `get_option_implied_volatility`、`get_option_underlying_detail`、
  `get_option_underlying_volatility`；
- 财务、因子、行业和宏观：`get_factor`、`get_fina_reports`、
  `get_industry_detail`、`get_industry_constituents`、`get_macro_detail`、
  `get_macro_tr`；
- 交易日历：`get_last_trade_date`、`get_trade_cal`。

“生产可用”只表示端点进入正式能力目录，不等于“交易可用”或“稳定性已确认”。
交易数据资格继续由数据类别、精确作用域和请求形状、运行时新鲜度、供应商发布状态、
数据质量及用户授权共同决定。研究、期货和期权数据不得因进入生产能力目录而获得执行权限。
`get_stock_detail` 保持一次验证的研究能力，不计入上述 26 项。基金端点及 SDK 标记为
“已废弃或未上线”的端点保持禁用；其余白名单外端点明确为未验证，不得回退调用。

### 8.2 补充数据源

系统可以接入经过批准的新闻、公告、财务和宏观数据源。每个来源必须记录许可证/使用权限、覆盖市场、更新时间、字段契约和失败状态。Agent 不得自行访问未批准来源。

行情、主数据及 PandaData 已覆盖的结构化市场字段以 PandaData 为唯一前端事实源。补充来源只进入研究证据层，不得作为前端行情的静默替代。

### 8.3 更新与新鲜度

- 各市场收盘数据可用后更新日线数据；
- 每条记录保存交易日、供应商时间和摄取时间；
- 新鲜度阈值按市场、数据类型和工作流配置；
- 超过阈值时，依赖产物自动 `expired`；
- 行情、授权、持仓、规则或依赖产物版本变化时，计划和草稿重新计算。

### 8.4 证据契约

| 字段 | 要求 |
|---|---|
| 来源 | 供应商、公告、新闻或确定性计算器的稳定标识。 |
| 时点 | 事件时间、数据时间和系统摄取时间。 |
| 适用范围 | 市场、资产、周期、币种和使用限制。 |
| 置信度 | 来源质量、覆盖率与计算状态共同形成；不得只由 Agent 自报。 |
| 事实/推断/建议 | 三类内容必须分开存储和展示。 |
| 未知项 | 数据缺口、冲突和无法验证的假设。 |
| 失效条件 | 数据、市场、策略或规则何时使结论不再适用。 |

### 8.5 数据失败规则

上游返回空数据、缺列、不完整行、不支持类型或权限错误时，调用明确失败。系统不得使用模拟值、最近值或其他市场数据静默补齐。

## 11. 状态模型

### 11.1 交易计划状态

`draft` → `pending_review` → `confirmed` → `executing` → `partially_completed` / `completed`

任一关键输入变化可进入 `expired`；用户可进入 `rejected` 或 `cancelled`。已确认计划不可原地覆盖。

### 11.2 订单草稿状态

| 状态 | 定义 | 可用操作 |
|---|---|---|
| `draft` | 草稿可编辑，尚未进入最终复核。 | 编辑、删除、进入复核。 |
| `pending_review` | 摘要冻结，等待用户最终确认。 | 返回编辑、处理软风险、确认。 |
| `confirmed` | 用户已确认，等待创建执行订单。 | 等待提交结果，禁止再次确认。 |
| `expired` | 行情、净值、授权、持仓、规则或正式风控已变化。 | 返回编辑并重新计算。 |
| `cancelled` | 用户放弃草稿。 | 查看，不可恢复为原草稿。 |

### 11.3 交易所订单状态

| 状态 | 定义 | 可用操作 |
|---|---|---|
| `submitting` | 请求正在发送。 | 等待，禁止重复提交。 |
| `unknown` | 无法确认请求结果。 | 查询状态，禁止重下。 |
| `accepted` | 已受理但未完全成交。 | 查看、符合条件时撤单。 |
| `partially_filled` | 部分成交。 | 撤余、等待。 |
| `filled` | 全部成交。 | 查看、复盘。 |
| `cancelling` | 撤单请求已发送。 | 等待。 |
| `cancelled` | 剩余数量确认撤销。 | 查看、创建新草稿。 |
| `rejected` | 被执行服务拒绝。 | 查看原因、创建修正版。 |
| `expired` | 有效期结束。 | 查看、创建新草稿。 |

### 11.4 场外基金状态

`draft` → `pending_review` → `submitted` → `accepted` → `pending_nav` → `confirming` → `confirmed` / `partially_confirmed`

合法撤销进入 `cancelled`，规则或数据失败进入 `rejected`。

### 11.5 风控状态

`checking` → `passed` / `confirmation_required` / `blocked` / `expired`

只有 `passed` 或经用户二次确认后的 `confirmation_required` 可以提交。`blocked` 绝不可覆盖。

## 13. 统一业务规则与异常处理

### 13.1 统一业务规则

| ID | 规则 |
|---|---|
| BR-01 | Agent 暂停只停止新 Agent 工作流，不自动撤销既有订单，也不阻止合法手动订单。 |
| BR-02 | 硬风控阻断不可由用户、Planner、治理 Agent 或其他服务覆盖。 |
| BR-03 | 冷静期禁止新增风险，只允许降低风险的卖出或赎回；该路径仍需用户确认和正式风控。 |
| BR-04 | 只有 `completed` 且 `trade_eligible=true` 的自动工作流产物可进入交易计划。 |
| BR-05 | `review_only` 和 `data_quality_review` 固定不能进入交易计划。 |
| BR-06 | 自动触发只能生成待审阅计划，不能创建订单草稿或提交订单。 |
| BR-07 | 手动订单可以绕过研究/策略工作流，但不能绕过正式风控和用户确认。 |
| BR-08 | 计划和草稿阶段只做约束、费用、影响和偏离计算；唯一正式 `RiskCheckResult` 在提交前生成。 |
| BR-09 | 修改草稿关键参数会使旧计算和正式风控过期。 |
| BR-10 | 修改未成交订单必须撤销原单并创建新草稿。 |
| BR-11 | 请求已发送、订单已受理和订单已成交是三个不同事实。 |
| BR-12 | 状态未知时禁止用同一草稿重复提交。 |
| BR-13 | 所有跨市场计算内部保留原币和汇率证据，界面统一展示人民币。 |
| BR-14 | 所有工作流、策略、计划、草稿、风控和订单均使用不可变版本。 |
| BR-15 | 任何失败不得静默降级为旧策略、默认结论、默认组合或看似成功的订单。 |

### 13.2 关键异常

| 异常 | 系统行为 | 禁止行为 |
|---|---|---|
| Agent 超时或错误 | 按错误类型在上限内重试；耗尽后失败并通知。 | 无限重试、隐藏错误或换旧结论。 |
| 数据陈旧/冲突/缺失 | 触发 `data_quality_review`，冻结受影响产物。 | 补造、拼接或使用最近值。 |
| 策略测试失败 | 工作流失败，不生成组合或正向交易结论。 | 治理 Agent 覆盖失败。 |
| 无可行组合 | 返回不可构造和冲突约束。 | 自动放宽授权或硬规则。 |
| 订单提交结果未知 | 标记 `unknown`，查询和对账。 | 重下相同订单。 |
| 仿真撮合异常 | 停止受影响撮合，保留既有事实。 | 伪造成交或回滚已确认历史。 |
| 公司行动来源冲突 | 暂停受影响标的更新并诊断。 | 选择任一来源静默入账。 |
| Agent 监控失败 | 单独显示 Agent 失败；确定性订单继续。 | 修改订单事实或撤销已受理订单。 |

## 14. 非功能需求

| ID | 类别 | 要求 | 验收标准 |
|---|---|---|---|
| NF-01 | 页面性能 | 正常网络下关键页面首屏 3 秒内可用。 | 1024、1440和1920 px 目标环境下达到要求。 |
| NF-02 | 交互性能 | 普通页面操作 1 秒内提供反馈。 | 超过时间时显示真实进行状态，不冻结主界面。 |
| NF-03 | 订单性能 | 草稿保存和提交请求 2 秒内返回明确状态。 | 超时进入可查询状态，不显示为已受理。 |
| NF-04 | 可用性 | MVP 月可用性不低于 99.5%。 | 按用户可用分钟计算，计划维护单独记录。 |
| NF-05 | 恢复 | RPO 不超过 15 分钟，RTO 不超过 2 小时。 | 演练可恢复账本、订单、工作流和审计关系。 |
| NF-06 | 容量 | 支持 100 个并发用户和 20 条并发工作流。 | 压测中不得产生重复订单、错配证据或状态污染。 |
| NF-07 | 安全 | 传输和静态存储加密，执行最小权限和敏感字段脱敏。 | 密钥不进入日志、研究产物或 Agent 上下文。 |
| NF-08 | 隔离 | 策略代码只在隔离工作区运行。 | 文件、网络、凭证和外部动作按白名单控制。 |
| NF-09 | 幂等 | 创建计划、保存草稿、提交订单和处理公司行动均具备幂等语义。 | 重复请求不创建重复事实。 |
| NF-10 | 浏览器 | 支持 Chrome、Edge和 Safari 最新两个主要版本。 | 核心交易旅程在各浏览器通过。 |
| NF-11 | 响应式 | 支持 1024–1920 px，1440 px 为设计基准。 | 小于 1024 px 显示桌面使用提示，不进入移动布局。 |
| NF-12 | 可访问性 | 以 WCAG 2.2 AA 为目标，关键流程支持键盘。 | 颜色不是唯一状态表达；焦点、标签和对比度通过检查。 |
| NF-13 | 数据留存 | 工作流与交易记录保留 3 年。 | 到期自动删除或不可逆匿名化，只保留合法最小审计记录。 |
| NF-14 | 失败可见性 | 所有外部依赖、Agent、数据和执行失败显示原因、影响范围和下一步。 | 不出现无原因的“成功”“暂无数据”或空白状态。 |
| NF-15 | 免责声明 | 仿真、Agent 建议和回测不构成收益保证或真实成交。 | 在研究、计划、复核、执行和复盘关键位置持续可见。 |

本文只定义产品对象与行为，不展开字段级 REST API 或事件协议。

## 15. 端到端验收场景

| 场景 | 前置条件 | 操作 | 期望结果 |
|---|---|---|---|
| AT-01 创建账户 | 有效上游授权 | 设置人民币初始现金 | 创建空仓仿真账户；不存在持仓导入入口。 |
| AT-02 自动研究 | 市场收盘数据可用 | 等待日常调度 | 运行市场/监控工作流；可生成待审阅计划，不创建订单。 |
| AT-03 公司只读研究 | 公司行情缺失但公告/财务可用 | 运行 `company_research` | 状态可为 `completed`，但 `trade_eligible=false`，不能生成计划。 |
| AT-04 数据陈旧 | 依赖数据超过阈值 | 启动市场工作流 | 切换数据质量诊断；旧产物过期。 |
| AT-05 用户暂停 | Agent 已暂停 | 启动自动工作流 | 不调用 Agent，生成阻断通知。 |
| AT-06 暂停后的手动订单 | Agent 已暂停，订单满足规则 | 创建并确认手动订单 | 不调用解释 Agent；正式风控通过后可以提交。 |
| AT-07 冷静期减仓 | 冷静期生效且已有持仓 | 创建卖出/赎回 | 允许进入正式风控；新增买入被阻断。 |
| AT-08 硬风险 | 正式风控命中硬规则 | 尝试提交 | 无覆盖和提交操作，订单不被执行服务受理。 |
| AT-09 软风险 | 正式风控命中软规则 | 查看原因并二次确认 | 保存确认记录后允许提交。 |
| AT-10 Agent 必选节点失败 | 研究 Agent 持续失败 | 运行自动研究 | 在上限内重试；耗尽后 `failed`，不生成默认结论。 |
| AT-11 手动解释失败 | 手动订单解释 Agent 失败 | 继续复核 | 显示失败；通过确定性规则后仍可提交。 |
| AT-12 策略测试失败 | Alpha 实现未通过测试 | 运行策略验证 | 工作流失败，不进入回测通过或组合构造。 |
| AT-13 无可行组合 | 授权约束互相冲突 | 构造目标组合 | 返回不可构造和冲突原因，不放宽规则。 |
| AT-14 重复提交 | 网络重试或重复点击 | 使用同一草稿提交两次 | 只创建一个订单。 |
| AT-15 日线限价部分成交 | 日线触价但参与率不足 | 执行限价单 | 生成部分成交和剩余数量，记录模型版本。 |
| AT-16 状态未知 | 提交返回不确定 | 再次点击提交 | 禁止重下，只允许查询和对账。 |
| AT-17 港美股卖空 | 可借数量和保证金满足 | 提交卖空订单 | 计提借券费和保证金；不支持市场不出现卖空。 |
| AT-18 场外基金申购 | 在截止时间前确认 | 提交申购 | 进入待净值/确认状态，最终按规则确认份额和费用。 |
| AT-19 公司行动 | 分红或拆并股生效 | 运行账本处理 | 只入账一次，更新现金/持仓并可追溯来源。 |
| AT-20 执行 Agent 失败 | 订单已被服务受理 | 监控 Agent 失败 | 订单继续按确定性服务处理，Agent 错误单独展示。 |
| AT-21 复盘 | 订单完成或计划结束 | 打开 T09 | 可对照研究、计划、修改、风险、成交、费用和组合影响。 |
| AT-22 三年到期 | 记录超过保留期 | 执行到期任务 | 自动删除或匿名化，只保留合法最小审计记录。 |
| AT-23 性能与容量 | 100 并发用户、20 并发工作流 | 执行核心旅程 | 满足性能目标，无重复订单、证据错配和状态污染。 |
| AT-24 浏览器与可访问性 | 支持浏览器和桌面宽度 | 完成研究到下单旅程 | 键盘可完成关键操作，状态不只依赖颜色。 |

## 16. 术语与范围外清单

### 16.1 术语

| 术语 | 定义 |
|---|---|
| Agent | 在受限输入、工具和权限内完成研究、辩论、解释或开发任务的执行单元。 |
| Planner | 选择正式工作流或创建受约束动态 TaskPlan 的产品级 Agent。 |
| 正式工作流 | 经过版本化治理的 15 条可复用业务流程。 |
| 动态 TaskPlan | Planner 在运行时基于批准能力创建的一次性任务计划。 |
| 确定性服务 | 对相同输入给出可复算结果，并掌握组合、资金、风险、订单和成交事实的服务。 |
| `trade_eligible` | 工作流产物能否进入自动交易下游的确定性资格字段。 |
| 正式风控 | 订单提交前唯一生成的 `RiskCheckResult`。 |
| 仿真交易 | 使用声明的数据、市场规则、费用、滑点和撮合模型产生的模拟结果。 |

### 16.2 范围外复核

以下能力不作为本 PRD 的验收内容：用户画像设计、实盘交易、期货/期权下单、已有持仓导入、移动端、文件导出、邮件/移动推送、Tick 数据、管理后台和字段级 API。
