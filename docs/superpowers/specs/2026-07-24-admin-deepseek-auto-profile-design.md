# Finance-God 独立管理端、DeepSeek 与自动画像设计

## 背景

当前管理端和用户端共用同源 `localStorage` 登录态。普通用户登录后，前端路由守卫会将管理路由重定向回用户端；后端同时通过 `role=admin` 强制 RBAC。用户端在未配置真实 API 时仍能对话，是因为开发环境默认启用了确定性 mock 文本适配器和浏览器语音能力。

现有画像对话把一次回答拆成“分析”和“用户确认”两次请求。AI 先写入待确认画像证据，用户确认后才更新画像并生成下一问。目标行为是不向用户展示内部分析确认：用户回答一次，系统自动分析、存储并进入下一问。

## 目标

- 同一浏览器中同时保持普通用户和管理员两套独立会话。
- 提供独立管理登录入口和仅限开发环境的测试管理员初始化方式。
- 接入 DeepSeek OpenAI-compatible API，默认使用 `deepseek-v4-flash`，支持管理员切换 `deepseek-v4-pro`。
- 每次回答后自动分析并持久化画像证据，直接生成下一问。
- 面向金融小白采用生活化、渐进式、单问题引导，不把访谈做成知识考试。
- 保留服务端确定性安全边界、请求幂等、审计和清晰的失败恢复。

## 非目标

- 本轮不引入独立管理子域、HttpOnly Cookie 或 CSRF 基础设施。
- 不支持管理员配置任意 API Base URL 或任意模型名称。
- 不实现 Flash 与 Pro 之间的自动故障回退。
- 不向用户展示画像置信度、内部维度代码或模型推理过程。
- 不在代码、数据库、API 响应、审计记录或日志中保存 DeepSeek Key 明文。

## 1. 独立管理端鉴权

### 前端路由与状态

- 新增 `/admin/login`，保留 `/admin/ai-settings`。
- 用户端继续使用 `finance-god-token` 和 `finance-god-user`。
- 管理端使用 `finance-god-admin-token` 和 `finance-god-admin-user`。
- 增加独立管理 Pinia store 和管理 API client。管理 client 只读取管理 token；普通 client 只读取用户 token。
- 两个 client 分别处理 401：用户 API 失败只清理用户会话并跳转 `/login`；管理 API 失败只清理管理会话并跳转 `/admin/login`。
- 普通用户访问管理路由时进入管理登录页，不再重定向到用户画像页。已登录管理员访问 `/admin/login` 时进入管理设置页。

### 后端合同

- 新增 `POST /api/v1/auth/admin/login`，请求字段与普通登录一致。
- 只有 `status=active` 且 `role=admin` 的账号能获得管理 token；普通用户凭据返回 403，错误信息不泄露账号是否存在。
- 管理 API 继续使用 `require_admin`，前端状态隔离不代替服务端授权。
- JWT 可沿用当前签名与载荷格式；每次管理 API 调用仍从数据库读取用户角色和状态，避免仅信任客户端角色。

### 开发测试管理员

- 新增幂等初始化脚本，只允许在 `APP_ENV=development` 下执行。
- 默认测试邮箱为 `admin@finance-god.local`。
- 密码通过本机 `.env` 中的 `DEV_ADMIN_PASSWORD` 提供；脚本拒绝空值、弱密码和非开发环境。
- 重复执行时更新该开发账号的密码、角色和 active 状态，不创建重复记录。
- 本轮在本地开发数据库执行脚本，并在交付信息中提供实际测试凭据；密码不提交 Git。

## 2. DeepSeek 配置与适配器

### 管理配置

- 文本 Provider 使用受控枚举：开发环境可选 `mock` 或 `deepseek`，其他环境只能使用 `deepseek`。
- DeepSeek Base URL 固定为 `https://api.deepseek.com`，不接受运行时任意 URL，降低 SSRF 和误配置风险。
- 模型使用受控枚举：`deepseek-v4-flash`、`deepseek-v4-pro`。
- 默认模型为 `deepseek-v4-flash`；管理员可显式切换 Pro。
- 配置继续使用版本号和 `AdminAuditRecord` 留痕。审计记录只保存 provider、model、启用状态、轮次和 prompt 版本等非敏感字段。
- API key 引用固定为或受控设置为 `DEEPSEEK_API_KEY`。管理 API 只返回 `api_key_configured: boolean`。

### 服务端密钥

- DeepSeek Key 仅存在于服务端进程环境变量 `DEEPSEEK_API_KEY`。
- 管理页面不显示、读取或修改 Key 明文。
- 本地开发可在被 Git 忽略的 `.env` 中设置该变量；生产环境由部署平台的 secret manager 注入。
- 应用日志、异常响应和连接测试结果不得包含 Authorization header、Key、完整供应商响应头或请求体中的敏感用户内容。

### OpenAI-compatible 调用

- 新增 DeepSeek text provider，通过异步 HTTP 客户端调用 `/chat/completions`。
- 设置明确的连接、读取和总超时；供应商 401/403、429、5xx、超时和网络错误映射为稳定的内部错误码与中文消息。
- 请求包含系统 prompt、必要的最小会话上下文和当前回答，不发送与当前分析无关的数据。
- 要求模型返回结构化 JSON，并用 Pydantic 校验以下字段：
  - `reply`
  - `target_dimension`
  - `profile_value`
  - `confidence`
  - `should_continue`
  - `end_reason`
  - `next_question`
  - `next_question_dimension`
- 响应缺字段、类型错误、越界值、未知维度或不合规文本时，不更新会话状态。

### 运行时选择

- 创建或恢复会话时固定 provider、model 和 prompt 版本，保证一份画像在同一配置快照下完成。
- 管理员后续切换模型只影响新会话，不在活动会话中混用模型。
- 选择 DeepSeek 后调用失败不得自动降级到 mock 或另一模型。
- Mock 仅在开发环境且被管理员明确选择时使用，用户界面显示“模拟 AI”状态。
- 生产环境缺少已启用的 DeepSeek 配置或 Key 时禁止开始 AI 访谈，并返回可操作的中文配置提示。

## 3. 自动画像状态机

### 单次回答的数据流

一次 `POST /onboarding/sessions/{id}/messages` 执行以下流程：

1. 校验用户归属、会话状态、当前问题、输入长度和 `request_id`。
2. 读取会话固定的 provider、model、prompt 和确定性约束状态。
3. 调用 AI adapter 得到结构化分析结果。
4. 服务端校验目标维度、置信度、画像值、追问次数、轮次、敏感维度和下一问内容。
5. 在一个数据库事务中写入用户消息、合并画像证据、更新分数/完整度/追问次数/轮次、写入助手消息和保存下一问。
6. 返回最新会话、用户消息、助手消息和本轮公开回复。

同一 `request_id` 重试必须返回首次成功结果，不重复增加轮次或画像证据。

### 移除确认流程

- 删除消息请求中的 `confirm_pending` 分支。
- API 会话响应不再公开 `pending_profile_evidence`。
- 前端删除 `confirmEvidence`、确认请求重试状态和 evidence panel。
- 模型 prompt 不再声称“确认后更新画像”，改为自然承接回答并提出下一问。
- 新数据库迁移移除待确认字段。旧活动会话中的待确认值不自动并入画像，以免存储用户尚未认可的历史推断；迁移清除该临时状态并保留当前问题。

### 失败与恢复

- AI、网络或响应校验失败时，不增加轮次、不更新画像、不替换当前问题。
- 前端恢复用户刚才的输入，并使用相同 `request_id` 重试。
- 服务端错误响应使用中文可操作文案，不暴露堆栈、密钥引用或供应商原始响应。
- 持久化事务失败时回滚本次全部业务写入，避免消息、画像和下一问不同步。

## 4. 新手友好访谈

### 提问原则

- 一次只问一个问题，每个问题只验证一个画像维度。
- 优先使用生活场景、时间需求、真实行为和直观选择，不直接询问夏普率、久期、波动率等术语。
- 不把“不懂”解释成低风险承受能力，也不把产品知识当成风险偏好的替代指标。
- 对无经验用户给出简短背景和生活化选择；对表达出实际经验的用户才逐步增加细节。
- 回答含糊时用场景或二选一追问，同一维度仍最多追问两次。
- 敏感收入问题明确可跳过，拒答保持中性，不降低评分。

### 确定性服务端边界

- 服务端根据客观资料中的投资经验、年龄、资金期限和历史回答向模型提供难度指示。
- 服务端校验下一问属于允许维度，且未超过追问和总轮次上限。
- 禁止具体产品推荐、收益承诺、保证本金或诱导性风险表达。
- 未成年人只得到金融教育与风险认知问题，不生成可执行投资方向。
- AI 候选问题不合规时本轮失败并可重试，不将未经校验的文本写入 `current_question`。

### 前端体验

- 页面只展示对话、当前问题、必要进度和敏感问题跳过入口。
- 不展示内部置信度、画像维度代码、评分增量或“确认判断”。
- 成功提交后自动滚动到最新助手回复和当前问题。
- 输入、按钮和状态文案保持面向普通用户的中文表达。

## 5. 管理端界面

- 管理登录页视觉上属于管理工作区，但沿用现有设计令牌与表单组件。
- AI 设置使用 provider 和 model 选择控件；Base URL 只读展示。
- 密钥区域只显示 `DEEPSEEK_API_KEY` 的配置状态和部署提示，不出现明文输入框。
- “测试连接”执行真实最小请求，并显示已选 adapter/model 以及成功或分类错误。
- 保存配置与测试连接是独立操作；测试失败不自动保存，保存也不假定连接成功。
- 管理页提供退出管理端操作，只清理管理会话。

## 6. 测试与验收

### 后端自动化测试

- 管理专用登录成功、普通用户拒绝、inactive 管理员拒绝和无效密码。
- 管理 API 对用户 token 返回 403，对管理 token 正常工作。
- 配置响应、审计和日志不包含 Key 明文。
- DeepSeek adapter 请求格式、Flash/Pro 选择、结构化响应解析和超时配置。
- 401/403、429、5xx、超时、网络失败、非法 JSON、未知维度和越界置信度。
- 一次回答自动写入画像并推进下一问，无 pending 状态。
- 失败不改变画像、轮次和当前问题；相同 `request_id` 重试不重复写入。
- 新手与有经验用户的 prompt 难度指示、敏感跳过、追问次数和未成年人边界。
- 数据库迁移 upgrade/downgrade 与 OpenAPI 合同。

### 前端自动化测试

- 用户与管理 token 使用独立 storage key 和独立请求 client。
- 两端登录、退出和 401 互不影响。
- 普通用户打开管理路由进入管理登录页，管理员登录后进入设置页。
- Provider/model 为受控选择，密钥明文输入不存在。
- 对话页没有确认面板，回答后直接显示下一问。
- API 失败恢复输入并复用请求编号；成功后自动滚动。
- 新手友好文案、敏感跳过和 mock 状态标识。

### 浏览器验收

- 在同一浏览器上下文同时登录普通用户和开发管理员，两套会话均保持有效。
- 管理端读取配置、选择 Flash/Pro、测试连接和保存配置。
- 普通用户完成客观资料后执行至少一轮真实 DeepSeek 访谈，验证回答、画像存储和下一问完整闭环。
- 检查浏览器控制台、后端日志和 API 响应不出现 Key。
- 在桌面和移动视口验证对话自动滚动、输入恢复和文本不重叠。

## 7. 安全与运维

- 用户提供的现有 DeepSeek Key 仅用于本地验收，不写入版本库；因其已出现在聊天记录中，正式上线前必须轮换。
- 生产部署必须通过 secret manager 注入 `DEEPSEEK_API_KEY`。
- 管理测试账号仅存在于开发环境；生产环境不得运行开发初始化脚本。
- DeepSeek 出站请求使用固定 HTTPS origin，并对响应体大小和超时设限。
- 用户回答属于敏感画像数据，日志默认不打印 SQL 参数、完整 prompt 或模型原始响应。

## 完成标准

- 普通用户登录不会阻止同一浏览器中的独立管理员登录。
- 管理端能真实验证并启用 DeepSeek Flash 或 Pro，且 Key 不泄露。
- 用户回答一次后无需确认，画像和下一问一致地更新。
- 新手能理解全部问题，不需要金融专业知识才能完成访谈。
- 所有新增和回归测试通过，真实浏览器流程和 DeepSeek 调用验收通过。
