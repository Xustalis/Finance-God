# Finance-God 开源项目与页面形态参考（历史研究）

> 历史研究记录：开源项目能力调研仍有效；其中移动端和旧导航建议不属于当前范围。
> 当前产品边界以 [前端统一设计规范](../../page-design/01_前端统一设计规范.md) 为准。

## 1. 文档目的

本文记录 [前端设计文档索引](../../page-design/00_前端设计文档索引.md) 中 T00–T10
页面的历史参考来源，回答三个问题：

1. 哪些开源项目可作为页面形态、交易流程或底层组件参考；
2. 哪些能力应补入 Finance-God，放在哪个页面；
3. 哪些内容因产品定位、技术成熟度或许可证原因只能借鉴，不能直接复制。

本轮核验日期为 **2026-07-23**。证据优先使用项目官方仓库的 README、LICENSE、官方文档和最近提交；页面补充以“AI 辅助、用户确认、事实可追溯”的产品边界为前提。

## 2. 结论

Finance-God 不应复刻单一开源交易终端，而应组合四类成熟模式：

| 参考层 | 主要项目 | 对 Finance-God 的价值 | 采用方式 |
| --- | --- | --- | --- |
| 组合与账户视图 | Ghostfolio | 多账户、资产表现、风险与组合构成 | 借鉴信息架构，补 T01/T08 |
| 交易复盘 | TradeNote | Dashboard、日视图、日历、标签、日志和截图 | 借鉴任务流，补 T09 |
| 执行与策略运维 | FreqUI、Hummingbot Dashboard | 健康状态、实时消息、配置、回测、部署和实例管理 | 借鉴状态模型，补 T00/T04/T07 |
| AI 研究与证据 | TradingAgents、FinRobot、AI Hedge Fund | 多角色研究、正反辩论、风险经理、组合经理、确定性计算与来源追踪 | 借鉴 Agent 分工，补 T03/T04/T10 |
| 数据接入 | OpenBB Open Data Platform | 数据提供商解耦、统一输出、多消费端 | 借鉴架构，补 T02/T03/T10 |
| 图表组件 | TradingView Lightweight Charts | 高性能交互金融图表和插件扩展 | 评估代码复用，补 T03 |

历史布局推导见 [页面布局辩论与裁决](页面布局辩论与裁决.md)。现行页面结构只以统一规范
和页面规格为准。

## 5. 各页面的具体补充

### 5.1 T00 交易授权与账户控制

参考 Hummingbot 的“账户—连接器—凭据”层级和 Freqtrade 的健康接口概念，新增：

- `ConnectionHealth`：连接状态、最后成功请求、最后同步、时钟偏差、环境和 Trace ID；
- “权限摘要”：只读/交易、允许市场、允许订单类型、IP/回调限制和到期时间；
- “连接测试”只校验可达性与权限，不产生订单；
- 纸面账户与实盘账户使用不同凭据对象，模式切换不能复用未明确授权的连接；
- 凭据只显示别名和末尾掩码，不回显完整 API Key、Secret 或 MFA 因子。

不采用 Hummingbot 示例中的默认账号密码设计；任何默认凭据都不得进入 Finance-God。

### 5.2 T01 交易总览与决策收件箱

参考 Ghostfolio 的组合概览和活动记录，新增：

- 组合表现切换：今日、周至今、月至今、年初至今、1 年、最大区间；
- 多账户聚合时同时展示“聚合总览”和账户级数据时点，禁止把不同步快照伪装成同一时点；
- 最近活动统一显示授权、计划、订单、成交、导入、复盘和连接异常；
- 系统健康只显示影响用户任务的项目，不将运行日志直接铺到总览；
- 收益卡不高于 P0/P1 决策待办，避免将产品导向频繁交易。

### 5.3 T02 自选与候选

参考 OpenBB 的 Provider 解耦思想，新增：

- 每个事实字段附 `provider_id`、数据时点、市场时区和延迟等级；
- 候选筛选器可保存为命名视图，但筛选器只保存条件，不保存一份独立候选事实；
- Provider 切换后旧研究进入“来源已变化/需复核”，不静默混合不同口径；
- 缺失数据、来源冲突和资产标识映射失败分别展示；
- 候选排序允许使用组合适配、成本、流动性、风险和证据完整度，不显示无法拆解的总分。

OpenBB 开源仓库主要是数据基础设施；企业 Workspace 的界面不能被当成该仓库可直接复用的开源 UI。

### 5.4 T03 AI 辅助交易台

参考 Lightweight Charts、TradingAgents 和 FinRobot，新增：

- 图表层使用独立图层注册表：行情、指标、AI 计划、持仓、订单、成交；每层有图例、来源和时点；
- AI 计划图层必须由结构化 `TradePlan` 字段生成，不解析自然语言坐标；
- 研究态展示“基本面/技术面/新闻与情绪/组合适配”章节摘要，并持续显示反方证据和未知项；
- 估值、仓位、手续费和风险数值由确定性计算器产生；LLM 只负责解释，不生成最终数值事实；
- 图表交互变化只更新研究上下文，不自动修改订单草稿；
- 图表组件技术选型若采用 Lightweight Charts，需落实 TradingView 署名、NOTICE 和插件兼容性检查。

本页不引入“AI Agent 一键下单”或多机器人控制台；研究结论仍通过订单草稿和最终复核层进入执行。

### 5.5 T04 交易计划详情

参考 TradingAgents 的多空辩论、FinRobot 的 Bull/Bear/Judge 和 Hummingbot 的配置—回测—部署顺序，新增：

- `ResearchDebate`：支持理由、反方理由、共识、未解决分歧和引用来源；
- `RiskOpinion`：波动、流动性、集中度、尾部风险和硬边界检查；
- `PortfolioDecision`：批准、退回、等待、数据不足；它是计划提案状态，不是用户交易确认；
- 每次参数修改形成配置版本，并保存修改前后的计划后组合差异；
- P1 增加回测快照：时间范围、成本假设、样本外标识、最大回撤、成交假设和数据版本；
- 回测结果只能作为证据，不能将“回测通过”直接转换为可提交订单。

AI Hedge Fund 的人物化投资流派不作为默认 UI。Finance-God 只保留可验证的分析职责，避免用人物权威替代证据。

### 5.6 T07 执行中心

参考 Freqtrade REST/WebSocket 状态和 Hummingbot 多实例监控，新增：

- 顶部连接条：执行服务、经纪商连接、事件流、最后回报和账户同步状态；
- 订单详情时间线：草稿确认、请求发出、服务受理、外部受理、部分成交、成交/撤单/拒单；
- 实时事件中断时进入“状态未知”，恢复后执行订单对账，不自动重复提交；
- 提供“暂停创建新订单”，但已提交订单仍按其事实状态跟踪；
- 用户可查看经过脱敏的错误码、外部回报编号、Trace ID 和重试记录；
- 批次计划显示已完成、处理中、失败和未开始的动作，不用单一进度条掩盖部分失败。

Freqtrade/Hummingbot 面向机器人和加密交易；Finance-God 借鉴其状态模型，不照搬 `start/stop bot`、强制入场或一键实盘部署。

### 5.7 T08 组合与偏离

参考 Ghostfolio 的多账户、表现区间和风险分析，新增：

- 组合表现与资产配置使用相同账户范围和估值时点；
- “当前、目标、计划后”三组配置可切换绝对金额、权重和风险贡献；
- 风险扫描至少覆盖集中度、资产/行业/地区分布、现金边界和数据完整度；
- 活动导入后先进入待核对状态，资产映射、币种和费用确认后才更新组合事实；
- 多账户聚合明确内部转账和重复活动，避免把资金移动计入收益；
- 组合页只创建 `TradePlan`，不直接产生订单。

### 5.8 T09 交易复盘

参考 TradeNote 的 Analyze/Reflect 双层结构，新增：

- 分析层：总览、日视图、日历、时间/星期/持有时长、交易与执行维度；
- 反思层：交易日记、Playbook、标签、错误标签、截图和用户自评；
- 价格图可叠加入场、退出和计划区间，但必须使用真实 `Order`/`Fill`；
- 同时记录结果指标与过程指标：是否遵守计划、是否越过确认、数据是否过期、滑点和费用；
- 用户可标记“好过程/坏结果”与“坏过程/好结果”，系统不得只按盈亏评分；
- 截图和用户原文作为不可覆盖附件，AI 摘要与原文分开保存。

TradeNote 偏日内交易，Finance-God 的日历和标签需同时支持长期持有、基金调仓和多日计划。

### 5.9 T10 过程与证据

参考 TradingAgents、FinRobot 和 OpenBB，新增：

- Agent 拓扑按职责展示：数据、分析、建模、综合、正方、反方、风险、组合裁决；
- 每个节点展示输入版本、输出 Artifact、开始/结束时间、状态、错误和父子关系；
- 增加检查点与恢复记录，明确“从哪个版本恢复”，不能把重新运行伪装成原运行连续成功；
- 数值血缘区分“Provider 原始事实—清洗字段—确定性计算—LLM 解释”；
- Provider、模型、Prompt/策略、计算器和规则均保留版本；
- 普通模式展示业务证据，高级模式展示技术元数据；两者都不暴露密钥、完整系统提示词或其他用户数据。

本地 [verifolio-unified-agents-0.2.0 第三方说明](../../../backend/vendor/verifolio-unified-agents-0.2.0/agent_framework/THIRD_PARTY_NOTICES.md) 已记录 TradingAgents、FinRobot 与 QuantSkills 的受约束适配。它可作为 Agent 目录和运行证据来源，但其 `proposed_actions` 仍只是待审核请求，不得成为订单事实。

### 5.10 M01 移动交易待办与确认

参考 Ghostfolio 与 TradeNote 的移动/PWA 经验，新增：

- 首页只保留 P0/P1 待办、账户模式、关键风险和最近订单状态；
- 复核页采用分步摘要，但账户、方向、数量、价格、费用、最大损失、计划版本和过期状态始终可见；
- 研究报告、复杂回测和多列证据转到 Web，同一对象通过 ID 与版本接续；
- 弱网时只允许保存本地未提交输入；最终提交必须在线重校验；
- 推送通知只打开待办，不视为确认或成交。

## 6. 新增对象与状态

| 对象/字段 | 单一事实源 | 使用页面 | 说明 |
| --- | --- | --- | --- |
| `ConnectionHealth` | 账户连接服务 | T00、T01、T03、T07 | 连接、最后成功、延迟、错误与 Trace ID |
| `DataProvenance` | 数据接入服务 | T02、T03、T04、T08、T10 | Provider、时点、口径、字段映射和质量 |
| `ResearchArtifact.debate` | 研究产物存储 | T03、T04、T09、T10 | `ResearchArtifact` 的结构化字段：支持、反方、分歧、共识和引用 |
| `ComputeArtifact` | 确定性计算服务 | T03、T04、T08、T09、T10 | 输入、公式/计算器版本和输出 |
| `AgentRun` | Agent 运行时 | T01、T03、T04、T10 | 拓扑、检查点、重试、错误和产物 |
| `BacktestSnapshot` | 回测服务 | T04、T09、T10 | 数据版本、假设、指标和局限 |
| `JournalEntry` | 复盘服务 | T09 | 用户原文、标签、附件和 AI 摘要引用 |

以上对象只增加证据与运行状态，不替代既有 `TradePlan`、`OrderDraft`、`RiskCheckResult`、`Order` 和 `Fill`。

## 7. 产品与技术边界

### 可以直接进入设计的模式

- 账户连接健康、数据时点和事件流状态；
- 配置版本、回测快照和“配置后再提交”的顺序；
- 多空/风险/组合裁决的可追溯研究产物；
- 交易日历、标签、截图和过程评分；
- 图表分层、插件扩展和数据 Provider 解耦。

### 不直接采用的模式

- 机器人一键启动后自主实盘交易；
- 以著名投资者人格代替结构化证据；
- 把回测收益作为默认主指标或交易许可；
- 在总览中堆叠运行日志、调试信息和所有 Agent 对话；
- 将 OpenBB Workspace 等非本仓库开源 UI 误认为可复用代码；
- 复制 GPL/AGPL 项目的页面代码进入未完成许可证评估的产品仓库。

### 许可证执行规则

1. GPL-3.0/AGPL-3.0 项目当前只用于研究功能和交互模式；任何代码、样式或模板复用必须先做单独法律评估。
2. Apache-2.0/MIT 也不是“无条件复制”：保留版权、许可证、NOTICE 与修改说明。
3. Lightweight Charts 除 Apache-2.0 外，项目 README 还要求面向用户的 TradingView 署名与链接，技术选型验收必须覆盖。

## 8. 验收补充

- T00/T07 能分别显示账户连接和执行事件流健康；断连不会被误报为“无订单”。
- T03/T04/T10 中任何估值、仓位和风险数值都可追溯到确定性计算输入与版本。
- T04 的多空辩论包含未解决分歧，不以“裁判 Agent”替代用户确认。
- T07 在 WebSocket/回报中断时进入“状态未知”，恢复后先对账再允许重试。
- T08 多账户聚合不会把内部转账计入组合收益。
- T09 同时支持过程评价与结果评价，并保留用户原文。
