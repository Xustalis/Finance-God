## 产品范围与流程
- PRD 定义的唯一主闭环为：理解用户 → 画像/心智状态 → 投资授权书 → 市场环境/研究 → 组合构造 → 仿真执行 → 风控/复盘 → 更新画像与授权。
- MVP P0 目标极宽，覆盖画像、心智干预、持仓导入、研究、策略、组合、交易、风控、审计和复盘；以当前实现成熟度不适合一次性“全量重写”。
- PRD 明确要求 L0-L3 自主级别，但公开 MVP 默认仅应开放到 L2 仿真；L3 实盘必须受账户、地区、能力开关和独立授权共同约束。
- PRD 对 100% 风控阻断、100% 可追溯、0% 心智干预越权提出硬指标，重构验收必须覆盖这些负向路径，不能只验收页面可见。
- 当前前端信息架构大体沿用 PRD 一级模块，但没有把“待办/下一步”串成引导式流程，也没有把 Agent 的阻断、输入版本和责任边界完整呈现。

## 初步权限判断
- 登录态 JWT 和资源 `user_id` 过滤已经存在，但“身份认证”和“业务授权”尚未形成独立、可审计的权限模型。

- 投资产品不能只做页面隐藏；画像归属、授权书有效性、自主级别、订单状态、账户类型、能力开关和资源所有权都必须在服务端校验。
- 用户已选择首版角色为“普通投资用户 + 运营/风控管理员”。当前用户表没有角色字段，也没有管理员资源、后台路由或处置工作流。

## 后端闭环断点（静态证据）
- 仪表盘接口返回固定的心智、市场、风险和活动示例，没有查询数据库，因此登录后看到的是演示数据而非用户真实状态。
- 复盘服务中组合偏离、策略表现和部分建议为固定值；复盘仅写一条审计事件，完整复盘结果没有持久化，无法可靠回看或比较历史。
- 策略、组合、订单虽有版本字段，但订单创建把 `strategy_version` 固定写成 1，版本追溯链不完整。
- 数据库初始迁移直接调用 `Base.metadata.create_all/drop_all`，不利于生产级可审计迁移和逐版本回滚。
- 当前 User 只有账户状态，没有 role/permission；认证接口也未校验 `status` 是否允许登录。
- 订单服务包含归属、自主级别、冷静期、单笔限额、现金/持仓校验和状态机，是可复用基础，但规则结果大量由服务内手工构造，未证明与新 RiskEngine 统一。
- 已确认风控存在新旧接口冲突：`RiskRule` 新基类要求 `rule_name`、`rule_type` 和异步 `check -> RuleResult`，但 `cash_sufficient/cooldown/mandate_active` 仍实现旧的 `name`、同步 `check -> dict`。
- `RiskAgent` 调用 `engine.run_checks()` 时未 `await`，且仍按旧字典汇总结果消费；这与当前异步引擎返回 `list[RuleResult]` 不兼容。
- `ExecutionService` 的关键校验未调用统一 RiskEngine，而是重复实现并手工生成风险检查结果，导致规则注册表、Agent 风控和真实下单路径可能给出不同结论。
- 所有核心创建接口大量直接接收无类型 `dict`，缺少 API 级字段校验、OpenAPI 精确合同和可生成的前端类型。
- PostgreSQL 当前已运行且健康，可使用独立临时数据库进行无污染闭环验证。

## 后端闭环实测
- 已在独立数据库 `finance_god_codex_audit_20260723` 成功执行迁移并启动后端。
- 注册、画像保存、画像确认和 L2 授权激活四步均能成功。
- 画像接口对完整结构能计算 1.0 完整度并完成版本确认。
- 授权创建目前接受无类型字典；静态实现未校验所引用画像版本真实存在且已确认，也未限制公开 MVP 创建 L3，存在越过业务前置条件的风险。
- 策略生成在没有市场环境、研究证据和持仓的情况下仍成功，输出基于硬编码比例、场景和假设；这是演示级生成，不满足 PRD 的证据、数据质量和可追溯要求。
- 目标组合在空数据库中返回 HTTP 成功，但 `constructible=false`、资产 100% 为虚拟 CASH，原因为缺少 instruments 主数据。
- 仓库没有资产主数据 seed/fixture/管理入口；公开资产接口只有列表查询。因此按 README 从空库启动后，普通用户无法自行解除“可用资产池为空”，闭环必然停止在组合构造。
- 没有市场环境时组合服务生成一个随机 UUID 写入 `market_context_id`，但数据库未设置外键，因此形成无法追溯的悬空引用；同时仍允许返回 draft 组合。
- 资产列表接口当前未要求认证；资产和市场环境均没有管理员维护/同步接口，进一步证明运营后台能力缺失。
- 授权资产类型存在大小写合同漂移：模型/组合逻辑约定 `etf`，但接口无枚举校验，传入 `ETF` 会静默形成空资产池。
- 组合算法读取了 `bond_ratio`，却没有构造债券桶或区分债券资产；生成权重时只加入权益和现金，再整体归一化，导致策略声明的债券配置丢失。
- 为继续诊断，已仅在临时库插入两条 `etf` 主数据，并成功生成使用小写类型的第二版 L2 授权和策略。
- 补充主数据后可执行策略 → 组合 → 订单 → 仿真成交，但组合实际权重约为 85.7% 权益 + 14.3% 现金，与策略声明的 60% 权益 + 30% 债券 + 10% 现金不一致。
- 成交成功后 `/holdings/current` 仍返回 `null`：执行服务更新的是 `SimulatedAccount.positions`，用户组合/交易页读取的是 `HoldingSnapshot`，两个资产口径没有同步或统一查询入口。
- 订单记录将 `strategy_version` 固定为 1，实测使用第二版策略生成的订单仍返回版本 1，追溯错误得到验证。
- 复盘实测虽然识别到 1 笔成交，但组合偏离与收益仍返回固定样例；滑点把金额直接乘以 10000 得到 1,071,250 bps，计算维度错误。
- 仪表盘在用户已经完成画像、授权、组合与成交后仍提示“完善投资画像”，证明它完全没有反映真实工作流状态。
- 因此当前只能称为局部 API 可运行，不能称为用户业务闭环成立。
- RiskAgent 实测报告 `healthy`、`success`、`checks_run=0` 和 `passed=true`，同时运行日志明确出现 `coroutine 'RiskEngine.run_checks' was never awaited`。
- 这意味着风险 Agent 当前会在实际未执行任何规则时错误放行，是 P0/阻断级问题；健康检查本身也无法识别该故障。
- 注册时还出现 passlib 读取 bcrypt 版本属性失败的运行时告警；现有 `bcrypt<5` 约束不足以消除当前兼容问题，需要统一密码哈希依赖或迁移实现。
- 临时审计后端已正常停止，独立数据库 `finance_god_codex_audit_20260723` 已删除并验证不存在；现有开发数据库未被修改。

## Vue 前端首轮静态复核
- `frontend/index.html` 仍使用旧 React 容器 `#root`，但 Vue 入口挂载 `#app`；这会造成生产页面空白，构建本身无法发现，必须增加入口 DOM 回归测试。
- 前端 `ApiEnvelope` 将 request id 声明为顶层字段，后端实际合同是 `meta.request_id`；虽然当前解包未读取该字段，公开类型仍需对齐。
- 导师图像目前用运行时 `/src/...` 字符串引用，生产构建不会自动打包该资源；补充 WebP 后必须改为 Vite 可追踪的资源引用并在构建产物中验证。
- `confirmEvidence` 每次调用都会生成新 UUID；如果服务器已提交而响应丢失，用户重试会变成新请求并得到冲突，未真正实现确认请求的幂等重放。
- 前端把 `liquidity_need` 也判断为可跳过敏感维度，但后端唯一可跳过维度是 `income_stability`，会导致前端展示一个必然返回 409 的操作。
- `restore` 捕获所有 current-session 错误后都尝试创建，会把 403/5xx 等真实故障误判为“无会话”；应只对 404 创建新会话。
- CSS 使用 `clamp(..., vw, ...)` 随 viewport 缩放字体，与本项目明确的稳定字号规范冲突，需改为断点控制。
- 对话首问在 UI 中询问投资目标，但后端当前维度为 `risk_tolerance`，会把回答提取到错误维度并污染画像。
- 报告页把消息发给当前 `window`，而不是受控的父窗口/打开者，外部工作台实际收不到消息。
- 管理端 GET 不返回密钥引用；当前前端空字段提交 `null`，后端无条件覆盖，会在编辑其他字段时静默清除既有密钥引用。协议需改为“省略即保留、显式引用才更新”。
- 应用启动未调用 `/auth/me` hydrate，前端守卫完全信任 localStorage 中可修改的角色；需在挂载路由前完成服务端身份刷新。
- 报告对低置信度内部维度代码未做中文映射，中文首发页面可能直出 `investment_goal` 等内部标识。

## 第二轮全面核查：白盒候选问题
- 注册请求只约束密码长度；`display_name`、`base_currency`、`region` 没有去空格、长度或格式约束，超长值可能越过 Pydantic 并在数据库层变成 500，待动态复现。
- 异步数据库引擎直接使用 `echo=settings.app_debug`；开发默认开启 debug 时会把完整客观画像和对话参数写入 SQL 日志，存在敏感信息泄露风险，应将 SQL echo 独立且默认关闭。
- 方向选择采用“先清空全部 selected，再设置目标”的两步写法，数据库没有每个画像最多一条 selected 的唯一约束；并发选择不同方向时可能留下双选，待 PostgreSQL 并发验证。
- `get_current_user_id`、`get_current_user_with_db` 在仓库中没有调用方；空 `repositories`/`tests.e2e` 包和三项未引用运行依赖属于确定性清理候选。
- `/health` 仅静态返回 healthy，不验证数据库；可用于进程存活，但不能作为依赖就绪探针。
- 管理端通过 `os.getenv(api_key_ref)` 判断密钥是否配置；任意密钥仅由 Pydantic `.env` 读取、未注入进程环境时可能误报未配置，需要按实际部署方式验证。
- `passlib 1.7.4` 与当前 bcrypt 4.3 运行时会尝试访问已移除的 `bcrypt.__about__` 并打印 traceback；密码功能仍可工作，但依赖组合噪声大且未来兼容性不稳。

## 第二轮全面核查：动态复现结论
- PostgreSQL 实测：101 字符称呼能通过请求模型，最终触发 `varchar(100)` 截断并返回 500；注册输入校验缺口确认。
- 同一注册请求的开发日志完整打印邮箱、bcrypt 哈希、称呼、币种和地区；`APP_DEBUG=true -> SQL echo=true` 的隐私问题确认。
- 对两个不同方向执行 50 轮并发双请求，每轮最终均只有一条 `selected=true`。当前 bulk update 会取得推荐行锁并把写入串行化，未复现双选；仍可把数据库唯一约束作为防御性增强，但不列为当前缺陷。
- 真实手机流程在“先拒绝一次证据，再确认 4 个维度”后，第 5 次回答稳定返回 502。服务端用 `turn_count >= min_rounds` 认定 projected ready，而 Mock 根据 `round_count` 仍返回下一问，导致双方终止条件不一致。
- 长对话移动端没有自动滚动：实测 `.message-stream` 为 `scrollTop=0`、`clientHeight=357`、`scrollHeight=1391`，当前问题完全不可见；用户会停留在最早消息，却在下方直接看到输入框。
- 对话错误直接显示英文内部合同 `AI provider returned an invalid next profile question`，不符合中文首发和可恢复错误文案要求。
- 客观信息第一步标题“你希望如何被称呼？”实际选项是性别，语义不一致；称呼已在注册页单独采集。
- 浏览器控制台存在两个可预期但嘈杂的 404：缺少 favicon，以及首次恢复用 404 表示无活动会话；后者是协议分支，前者属于可清理的静态资源缺口。

## 2026-07-24 管理端与逐问画像需求初查
- 管理端和用户端当前共用同源 `localStorage` 的 `finance-god-token` / `finance-god-user`；普通用户已登录时，路由守卫会把 `/admin/ai-settings` 重定向到 `/app/exe?notice=admin_required`。这不是 Cookie 冲突，而是单会话、单角色的前端鉴权设计。
- `/login` 对任何已认证用户都会重定向，普通用户无法在同一浏览器上下文中切换到管理员身份；现有代码也没有独立管理登录入口。
- 开发环境未持久化 AI 配置时默认使用 `mock-structured-v1` 文本适配器，STT/TTS 默认使用浏览器 Web Speech API，因此用户端能完成“AI”对话并不表示已配置真实云端 API。
- 当前画像状态机把 AI 提取结果写入 `pending_profile_evidence`，前端要求用户确认或拒绝，确认后才进入 `profile_evidence` 并推进下一问。取消确认必须改后端原子写入/推进合同及前端交互和测试，不能只隐藏确认 UI。
- 后端管理 API 使用 `require_admin` 做真实 RBAC，不能靠单独打开页面绕过；前端路由拦截和后端 403 是同一权限规则的两层实现。
- 当前 `AIAdapterRegistry` 仅注册 `mock` 文本提供商以及 `browser` STT/TTS。管理页是配置与审计壳，但尚没有 OpenAI、DeepSeek、火山等真实文本/语音适配器；填写未注册 provider 会在保存或测试时失败。
- 要实现“问一个答一个、后台分析存储、直接生成下一问”，应让单次 message 请求原子完成：保存用户消息、执行提取、合并画像证据、更新维度/轮次/完整度、保存助手消息与下一问。这样可删除 `confirm_pending` API 分支和前端 evidence panel，同时保留 request_id 幂等和事务回滚。
- 用户确认管理端必须在同一浏览器中使用独立管理员会话，不能覆盖普通用户端登录态；需提供独立 `/admin/login` 和开发环境测试管理员账号。
- 用户确认真实文本模型使用 DeepSeek 的 OpenAI-compatible API；默认模型为 `deepseek-v4-flash`，管理端可切换到 `deepseek-v4-pro`，不做自动模型回退。API key 只以服务端环境变量引用，不持久化明文。
- 未配置真实模型时应区分环境：开发环境可明确标识并使用 deterministic mock；生产环境禁止启动 AI 访谈并提示管理员配置，不能把 mock 表现为真实 AI。
- 用户逐段确认了推荐架构：同源双会话独立管理登录；开发管理员由受控初始化脚本创建；DeepSeek provider/Base URL/Flash/Pro 使用受控选项；Flash 默认、Pro 手动切换、不自动 fallback。
- DeepSeek adapter 应一次返回结构化的回复、画像证据和候选下一问，但轮数、维度追问、敏感问题、未成年人和禁用收益承诺等约束必须由服务端确定性校验。
- 用户确认访谈面向可能完全没有金融知识的用户：一次只问一个生活化问题，根据客观资料和回答渐进调整难度；“不懂”不作为低风险能力证据；含糊回答用场景或二选一引导，同维度最多两次。
- 用户端不展示置信度和中间 AI 判断；回答成功后服务端自动分析并持久化，再返回下一问。前端应自动滚到最新问题，失败时保留输入并以同一 request_id 重试。
